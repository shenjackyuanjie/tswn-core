## 5. 结论

基于这次补跑的数据，当前可以比较明确地说：

- `0.2.20` 延续了上一轮的结果一致性收口；以 `tests/sqp5900.txt` 的 `3v3v3 + 4000 case` 口径看，`failed case = 0` 仍然成立。
- 当前工作树依然首先保持了结果一致性收口；以 `tests/sqp5900.txt` 的 `3v3v3 + 4000 case` 口径看，`failed case = 0` 仍然成立。
- 这轮继续把高频目标打分里的 `alive_group_containing(...).len()` 线性扫描改成了基于 team index 的 `alive_group_len_containing()`，在保持行为一致的前提下，继续压低了 action 路径开销。
- 相较 `0.2.14`，当前回退已经明显缩小：
  - 单线程两条样本慢约 `1.6%~2.3%`
  - 多线程两条样本慢约 `7.9%~10.7%`
- 单线程两条样本都已经重新快于 `opencode_branch`；
- 多线程里复杂样本已经追平并略快于 `opencode_branch`，`aaa` 仍有一段差距，当前优先级继续放在多线程剩余开销上；
- 因此当前更适合把 `0.2.14` 继续视作性能基线，把 `0.2.20` 视作“正确性稳定、性能继续回收中”的状态。
