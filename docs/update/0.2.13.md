# tswn_core 0.2.13 更新说明（胜率执行逻辑下沉到 core、CLI/CAPI/Python 统一复用）

> `0.2.13` 这一版的重点不是再给 `tswn-cli` 追加一层新命令，而是把已经分别散落在 `cli`、`capi`、`py` 三侧的 prepared win-rate 执行逻辑，正式收拢到 `tswn_core` 里统一维护。
>
> 这轮工作主要做了三件事：
>
> 1. 在 `tswn_core` 中新增统一的 `win_rate` 模块，收拢线程分发、seed 调度、胜负判定和 timing 汇总；
> 2. 让 `tswn-cli`、`tswn_capi`、`tswn_py` 全部改为复用同一套 core 实现，避免三边继续各修各的；
> 3. 顺手把 Python 高层接口补上可选 `thread` 参数，并同步步进 `tswn_capi` / `tswn_py` 版本与 changelog。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

这一轮的改动目标很明确，主要是解决“同一个功能被实现了三遍”的工程问题。

在本次重构前，prepared win-rate 相关逻辑实际上分散在三处：

- `tswn-cli` 自己维护了一套线程调度、seed 生成和胜负统计；
- `tswn_capi` 又复制了一套相同语义的 FFI 内部实现；
- `tswn_py` 也有一份自己的高层 helper 实现。

这种结构短期能工作，但有一个明显问题：

- 一旦某处 seed 调度修了 bug，另外两处还得再修一遍；
- 一旦线程默认策略想调整，也得三边同时同步；
- 时间一长，很容易出现 CLI、C-API、Python 三个入口在同一输入下行为细节逐渐漂移。

因此，`0.2.13` 的目标不是再扩新功能，而是把：

- `thread=0/1/n` 的统一语义
- JS profile seed 调度
- prepared runner 多线程 worker 分发
- 第一组胜率统计
- `init/fight` timing 汇总

这几块真正属于“核心算法行为”的内容，下沉回 `tswn_core`。

---

## 2. 主要改动

## 2.1 在 `tswn_core` 中新增统一的 `win_rate` 模块

涉及文件：

- `crates\tswn_core\src\win_rate.rs`
- `crates\tswn_core\src\lib.rs`

### 本次改动

这一版在 `tswn_core` 里新增了统一的 `win_rate` 模块，对外集中提供 prepared 胜率统计能力。

核心收拢内容包括：

- `thread` 参数语义统一：
  - `0`：自动线程数
  - `1`：单线程
  - `n`：指定线程数
- 自动线程数策略统一：
  - 继续沿用当前 CLI 的 `available_parallelism() * 5 / 4`
  - 再按总局数上限收敛
- JS profile seed 调度统一：
  - `win_rate rq` 模式下首局仍保持无 seed
  - 后续局数按 `seed:33554431@! + i` 递增
- `PreparedRunner` 多线程执行统一：
  - 使用 `Arc + AtomicUsize + std::thread::spawn`
  - 单线程和并行线程都走同一套种子语义
- 统计结果统一：
  - `wins`
  - `total`
  - `init/fight` timing

为了便于上层直接复用，本次还抽出了统一结果结构：

- `WinRateSummary`
- `WinRateTiming`

### 效果

这一步的价值主要不在“多快了多少”，而在于把真正的公共逻辑放回了公共层：

- 后续修 seed、调线程策略、改统计口径时，只需要改一处；
- CLI / C-API / Python 的胜率行为终于有了一个共同的实现源头；
- 后面如果还要补更多对外包装层，也可以直接复用 `tswn_core::win_rate`，不必再复制一份新实现。

---

## 2.2 `tswn-cli` 胜率相关路径改为统一调用 core 实现

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\bench.rs`
- `crates\tswn_core\src\bin\tswn_cli\fight.rs`

### 本次改动

本次把 CLI 里原本自己维护的 prepared win-rate worker 实现删除，改为直接调用 `tswn_core::win_rate`。

覆盖路径包括：

- `bench win-rate`
- `bench group-win-rate`
- `raw !test!` 中的两组胜率分支

CLI 自身保留的部分主要只剩：

- 参数解析
- 输出格式
- benchmark 的 perf 打印
- 原有评分 benchmark 路径

而真正的 prepared 胜率执行逻辑现在统一交给 `core`。

### 效果

这样处理后，CLI 的职责会更干净：

- 它不再需要自己维护一套“和 core 语义等价”的重复实现；
- 后续若 core 中的 win-rate 语义修正，CLI 会自动继承，不会再出现只修了一边的情况；
- `bench` 与 `raw !test!` 的 prepared 胜率执行路径也因此更统一。

---

## 2.3 `tswn_capi` 改为复用 `tswn_core::win_rate`

涉及文件：

- `crates\tswn_capi\src\lib.rs`
- `crates\tswn_capi\Cargo.toml`
- `crates\tswn_capi\CHANGELOG.md`

### 本次改动

这次没有再改 `tswn_capi` 对外暴露的胜率函数签名；`0.2.0` 中刚引入的 `thread: u32` 参数语义保持不变。

本轮真正变化的是内部实现：

- 删掉了 `tswn_capi` 自己维护的 prepared win-rate worker / seed / 线程分发逻辑；
- 改为在 FFI 层直接调用 `tswn_core::win_rate::prepared_win_rate(...)`；
- `tswn_capi` crate 版本从 `0.2.0` 步进到 `0.2.1`；
- changelog 中单独补充“内部实现改为复用 core”的说明。

### 效果

这意味着 `tswn_capi` 现在更像一层真正的 FFI 壳：

- 负责 UTF-8 输入、错误码转换和输出结构体写回；
- 不再在 FFI 层维护一套独立的核心胜率执行代码。

对调用方来说：

- `0.2.1` 相比 `0.2.0` 没有新的 API / ABI break；
- 但内部逻辑已经切到和 CLI / Python 完全一致的 core 实现源头。

---

## 2.4 `tswn_py` 高层接口补齐 `thread` 参数，并改为复用 core 实现

涉及文件：

- `crates\tswn_py\src\lib.rs`
- `crates\tswn_py\tswn_py\tswn_py.pyi`
- `crates\tswn_py\Cargo.toml`
- `crates\tswn_py\CHANGELOG.md`

### 本次改动

Python 这边本轮做了两件事：

#### 1. 给高层 helper 新增兼容的可选 `thread` 参数

新增后的签名为：

- `win_rate(raw, n, eval_rq=None, thread=0)`
- `group_win_rate(target, against, n, eval_rq=None, thread=0)`

参数语义与 CLI / C-API 对齐：

- `0`：自动线程数
- `1`：单线程
- `n`：指定线程数

因为这里采用的是可选参数形式，所以旧调用方式仍然兼容。

#### 2. Python 内部改为调用 core 的统一实现

原本 Python 自己那份 prepared win-rate 循环和 seed 调度逻辑也被移除，改为直接调用：

- `tswn_core::win_rate::prepared_win_rate(...)`

同时：

- `tswn_py` 版本从 `0.1.10` 升到 `0.1.11`
- `tswn_py.pyi` 也同步更新了新签名

### 效果

这一步的收益比较直接：

- Python 用户终于可以像 CLI / C-API 一样显式控制线程行为；
- 老代码不需要修改，兼容性风险较低；
- Python 高层 helper 的执行语义，也正式与 CLI / C-API 对齐到同一套 core 实现。

## 3. 版本与兼容性说明

这一轮涉及三个 crate 的版本状态：

- `tswn_core`：`0.2.13`
- `tswn_capi`：`0.2.1`
- `tswn_py`：`0.1.11`

其中需要特别说明的是：

- `tswn_core 0.2.13` 的主线价值是内部重构与统一实现来源；
- `tswn_capi 0.2.1` 相比 `0.2.0` 没有新增对外 API / ABI break，这次步进主要用于标记内部实现切换；
- `tswn_py 0.1.11` 属于兼容增强：新增了可选 `thread` 参数，但旧调用方式仍然成立。

换句话说，这一轮真正的“外部接口变化”主要落在 Python 的可选参数扩展上；而 core / capi 侧的重点则是把内部行为统一收拢，降低后续继续演化时的漂移成本。

## 5. 小结

`0.2.13` 这一版最核心的意义，不是“多了一个新命令”或者“多了一组对局规则”，而是把此前已经在三个入口层重复存在的 prepared win-rate 逻辑，真正拉回到了 `tswn_core` 这一层统一维护。

这件事的直接结果是：

- CLI / C-API / Python 三侧的胜率语义更一致；
- 后续再修线程策略或 seed 细节时，不需要三边重复跟进；
- Python 还顺手补齐了此前缺失的 `thread` 控制能力。

从长期维护角度看，这类“把重复实现收回公共层”的版本，往往比单次增加一个表面功能更重要，因为它直接决定了后续功能继续扩展时会不会越写越分叉。
