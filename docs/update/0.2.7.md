# tswn_core 0.2.7 更新说明（时序链、Charm / alive 顺序与召唤物生命周期继续对齐）

> `0.2.7` 是一版很典型的“最后几处顺序语义继续打磨”更新：重点不是新增系统，而是继续把 `post_action` / `post_defend` / revive / summon / charm 这些最容易造成目标链分叉的边界行为压向 JS。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

`0.2.6` 之后，剩下的 failed case 已经不多，但也更难：

- 不是简单的数值写错；
- 而是某个状态早一拍、晚一拍，或者某个刚复活/刚生成的对象在某一步到底算不算 alive。

`0.2.7` 主要收敛四类误差：

- `post_action` / `post_defend` 链的统一顺序仍与 JS 有残余偏差；
- Charm 的有效队伍、缓存队伍与 recharm 叠层判断还不完全一致；
- summon / clone / shadow / pending spawn 在命名、生命周期和日志上还有少量分叉；
- 剩余 case 的定位需要更大的样本和更细的回归覆盖。

所以这版可以视为一次“顺序链与运行时视图继续校准”的修复版。

---

## 2. 主要改动

## 2.1 `post_action` / `post_defend` 顺序继续统一到 JS 语义

涉及文件：

- `crates\tswn_core\src\player\impl_runtime.rs`
- `crates\tswn_core\src\player\skill.rs`
- `crates\tswn_core\src\player\skill\store.rs`
- `crates\tswn_core\src\player\state.rs`
- `crates\tswn_core\src\player\skill\act\iron.rs`
- `crates\tswn_core\src\player\skill\skl\protect.rs`

### 本次修正

- `Charge` 的 `post_action` 被拆成前后两段，模拟 JS 中尾钩子晚于部分 state tick 的顺序；
- `ProtectState` 记录注册时机，并按 skill/state 混合链的顺序进入防御链，而不是粗暴地后插；
- `post_defend` 不再分 skill 和 state 两段串行执行，而是合并成统一优先级链排序；
- 同优先级的 `post_defend` 项再以 `registration_order` 做稳定 tiebreaker；
- `IronState` 死亡后到期时的解除日志也补齐，并在战斗结束边界上按 JS 语义中断尾链；
- `ProtectState` 改用 `set_state_no_update`，避免注册时提前触发 `update_states()`，从而让 `Haste` 的蓄力加成过早生效；
- `Protect` 在 `merge/devour` 后即使 link 已存在，也会刷新 link 上记录的 level。

### 效果

这批改动的核心收益是：

- `Protect` / `Reflect` / `Iron` / `Charge` 这些高度依赖顺序的路径更接近 JS；
- 同层钩子不再因为实现拆层方式不同而走出另一条战斗分支；
- 铁壁、蓄力和守护相关的日志与时序更稳定。

## 2.2 Charm、全局 alive 视图与复活同步顺序修复

涉及文件：

- `crates\tswn_core\src\engine\world_state.rs`
- `crates\tswn_core\src\engine\engine_core.rs`
- `crates\tswn_core\src\engine\tick.rs`
- `crates\tswn_core\src\player\skill\act\charm.rs`

此前 Rust 对“当前全局还活着的人按什么顺序排列”与“recharm 时同队比较到底该用哪一个队伍视角”的理解，和 JS 仍有残差。

### 本次修正

- 引入 `WorldState.flat_alive`，维护与 JS `Engine.e` 更一致的全局存活顺序；
- 新实体插入、复活插回、死亡移除都改为显式维护这条全局 alive 链；
- `select_targets` 等依赖 `all_alive` 的路径开始以 `flat_alive` 为准；
- `CharmState.post_action_priority` 调整到与 `SlowState` 同层，让同层顺序回到注册时机决定；
- recharm 时不再更新 `effective_team_idx`，保留首次 charm 时缓存下来的目标选择视角；
- 同时新增 `source_team_idx`，在 recharm 判断“是否同队叠层”时使用已解析出的源队伍，而不是重新从原始 group 推导；
- 同 tick 的 revival 会优先于 pending remove 同步进 `WorldState`，并避免把仍然存活的对象错误删掉。

### 效果

这批改动直接修的是最容易让 FFA 后续目标链分叉的问题：

- `all_alive` 顺序更接近 JS；
- Charm 的“缓存视角”和“重魅惑比较视角”不再混用；
- 复活、死亡与下一步目标选择之间的顺序更稳定。

## 2.3 隐匿、召唤物、分身与日志一致性继续修复

涉及文件：

- `crates\tswn_core\src\player\skill\skl\hide.rs`
- `crates\tswn_core\src\player\skill\act\summon.rs`
- `crates\tswn_core\src\player\skill\act\clone.rs`
- `crates\tswn_core\src\player\skill\act\minion.rs`
- `crates\tswn_core\src\player\skill\act\shadow.rs`
- `crates\tswn_core\src\player\impl_runtime.rs`
- `crates\tswn_core\src\bin\tswn_cli.rs`
- `crates\tswn_core\src\bin\tswn_case_miner.rs`

### 本次修正

- `Hide` 开始统计同一 action 内 owner 自己刚生成、但尚未 sync 的 pending shadow；
- 同时又限制为只统计施法者自己的 pending shadow，避免把整队其他人的待生成实体错误算进去；
- 血祭 / summon 在重复施放时，如果旧 summon 已死亡，会复用原对象并通过 revival 刷新状态，而不是新建一只“同名新实体”；
- summon 技能补上 `boosted` 标记判断，保证重复施放与 boost 生命周期更接近 JS；
- minion 的命名规则继续统一：沿 root owner 共享 `?n` 编号，区分内部名、展示名和日志名；
- clone 的 display name override 修复，避免分身显示名再错位；
- shadow 使用稳定 `sort_int`，减少调度与显示顺序抖动；
- `post_action` 状态链中的 `alive` 参数改为逐项刷新；
- Reflect 的反伤顺序挪到 `kill/post_kill` 回调之后；
- pending enemy spawn 也会参与“敌方是否仍存活”的判断，避免误跳过吞噬等 `post_kill`；
- 战斗已结束后，combat minion 不再额外输出“被击倒了”，只保留“消失了”。

### 效果

这批修正解决的是“同一个实体在不同阶段到底被当成谁、算不算已进场、该不该继续输出日志”的问题。

对日志对账来说，价值很直接：

- summon / clone / shadow 的身份更稳定；
- pending spawn 不再把战斗结束与 `post_kill` 判断带歪；
- 战斗结束边界上的多余死亡文案被继续压掉。

## 2.4 数值边界、样本扩展与排查能力补强

涉及文件：

- `crates\tswn_core\src\player\skill\act\quake.rs`
- `crates\tswn_core\src\player\skill\act\accumulate.rs`
- `crates\tswn_core\src\bin\tswn_case_miner.rs`
- `crates\tswn_core\src\player\test.rs`
- `crates\tswn_core\src\player\test\skills.rs`
- `crates\tswn_core\src\engine\test\runner\mod.rs`

### 本次补强

- `Quake` 保留 JS 的原始浮点常量，不再把边界 case 的伤害从 23 压成 22；
- 聚气清除后的重用倍率生命周期继续对齐 JS；
- case miner 扩展到每模式 500 样本；
- 新增 / 整理 summon、charm、world sync、iron、hide 等回归测试；
- 对于尚未完全收敛的问题，补上 `ignore` 用例持续追踪，例如 merge 技能槽位不匹配时隐藏技能不继承的问题。

### 效果

`0.2.7` 的另一半价值，不在“又修了一个点”，而在“接下来还能继续更快地发现还差哪一个点”。

## 3. 影响 / 对齐点 / 兼容性

`0.2.7` 没有引入新的 Rust API breaking change，但行为变化依然明显：

- `post_action` / `post_defend` 的顺序会继续变化；
- Charm recharm、`all_alive`、revive 同步与目标选择会更接近 JS；
- summon / shadow / clone / minion 的命名、生命周期和日志会继续变化；
- `post_kill`、反伤、战斗结束边界上的输出会更干净。

如果你在做 SBY 对账或黄金日志校验，这一版会直接改变结果；
如果你只是从上层调用 crate，本次仍主要是行为层修复而不是接口层迁移。
