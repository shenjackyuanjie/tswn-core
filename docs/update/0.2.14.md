# tswn_core 0.2.14 更新说明（回迁性能优化、补齐跨版本性能对照）

> `0.2.14` 这一版的重点，是把已经在 `tswn_core_opencode` 分支中验证过的两组性能优化正式回迁回 `main`，并把 `run_to_completion()` 复用 `RunUpdates` 时的结果分叉风险一并修掉。
>
> 这轮改动主要有三件事：
>
> 1. 回迁 `PlayerStateStore` 的单表存储优化；
> 2. 回迁 `run_to_completion()` 的 `RunUpdates` 复用，并同步修复批次 `id` 复用问题；
> 3. 回迁 `Storage::player_group` 反向索引和战斗结束判定热路径去分配优化。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`  
> 长期性能记录页见：`performance.md`

---

## 1. 改动目标

这次不是新增玩法，而是把已经证明有收益、且不会改变对外接口的内部性能优化，正式带回主线。

目标很明确：

- 降低状态系统排序与遍历时的额外哈希查询；
- 降低批量模拟时 `RunUpdates` 的重复分配与详细帧缓存开销；
- 降低“判断战斗是否结束”这类高频路径上的 `Vec` 分配和队伍线性扫描；
- 同时确保 `run_to_completion()` / `bench win-rate` 和逐回合 `main_round()` 不会再因为内部复用策略出现行为分叉。

---

## 2. 主要改动

## 2.1 `PlayerStateStore` 改为单表存储

涉及文件：

- `crates\tswn_core\src\player\state.rs`
- `crates\tswn_core\src\player\impl_runtime.rs`

原先 `PlayerStateStore` 维护两张表：

- `states`
- `state_orders`

现在改为单个：

- `entries: HashMap<StateTag, StateEntry>`

每个 `StateEntry` 同时保存：

- `state`
- `order`

这样可以在保持注册顺序语义不变的前提下，减少：

- 状态排序时对 `state_orders` 的额外查询
- 状态遍历时的双表来回访问

这部分收益主要落在状态热路径本身，属于低风险、稳定型优化。

## 2.2 `run_to_completion()` 复用 `RunUpdates`

涉及文件：

- `crates\tswn_core\src\engine\engine_core.rs`
- `crates\tswn_core\src\engine\runners.rs`
- `crates\tswn_core\src\engine\update.rs`
- `crates\tswn_core\src\player\test\skills.rs`

这次把 `main_round()` 的内部逻辑抽成了：

- `main_round_into(..., &mut RunUpdates)`

然后让 `run_to_completion()`：

- 复用单个 `RunUpdates`
- 使用 `RunUpdates::new_no_capture()`
- 每轮先 `reset()` 再推进主循环

这样批量模拟时：

- 不再每轮重新创建新的 `RunUpdates`
- benchmark 高速路径不会再缓存详细事件帧

### 同步修复的 correctness 风险

如果 `reset()` 只清空内容，但不刷新新的 `updates.id`，那么像 `CounterSkill` 这类依赖批次 `id` 做去重的逻辑，就会把跨轮次事件误判成同一批次。

这次已经同步修掉：

- `RunUpdates::reset()` 现在会刷新新的批次 `id`

并补了最小回归测试：

- `counter_treats_reused_updates_as_new_batch_after_reset`

因此这次回迁不是“只要快，不管结果”，而是把性能收益和结果正确性一起带回主线。

## 2.3 `Storage::player_group` 反向索引与热路径去分配

涉及文件：

- `crates\tswn_core\src\engine\storage.rs`
- `crates\tswn_core\src\player\impl_runtime.rs`

这一版给 `Storage` 新增了：

- `player_group: HashMap<PlrId, usize>`

它由 `sync_groups()` 统一维护，用来表示：

- 玩家 `PlrId` -> 所属队伍索引

基于这层反向索引，下面这些接口不再需要每次扫描整张 roster：

- `group_index_of()`
- `group_containing()`
- `alive_group_at_team_of()`

同时新增：

- `iter_player_ids()`
- `iter_pending_spawns()`

然后把下面两条高频判定改成直接迭代：

- `has_alive_enemy_or_pending()`
- `is_battle_over()`

这样避免了原来反复构造 `Vec` 和反复线性扫描队伍的开销。

## 5. 小结

`0.2.14` 这一版真正做成了两件事：

- 把已经在 `opencode` 分支里验证过的性能收益，正式、可回归地带回了主线；
- 在这个基础上又继续补进了两条低风险热路径优化，同时把 `RunUpdates` 复用时最容易埋进去的 correctness 风险，用 `id` 刷新和回归测试一起兜住了。

从结果上看，当前主线已经可以同时满足：

- 比 `0.2.13` 更快
- 单线程与复杂样本多线程比当前 `opencode_branch` 也更快
- 且 `bench win-rate` 不再和逐回合路径发生结果分叉

`aaa` vs `bbb` 的 CLI 多线程还需要继续观察，但整体上，`0.2.14` 仍然可以作为一个新的、比较干净的主线性能基线来往下迭代。
