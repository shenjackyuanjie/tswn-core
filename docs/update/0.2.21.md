# tswn_core 0.2.21 更新说明（状态标签接口 + Accumulate/Hide 运行时态标识）

> `0.2.21` 这一版的核心工作是为 WASM 前端展示玩家实时状态标签铺路。
>
> 之前在 `tswn_wasm::fight::collect_states()` 中，每个玩家只有 `hp/mp/attack/defense/…` 等数值属性，前端只能看到"这个角色还剩多少血"，看不到"这个角色当前是不是在隐匿/聚气/狂暴状态"。
>
> 这轮在 `tswn_core` 侧暴露了两个关键接口：
>
> - `Player::skill_storage()`：让外部能遍历玩家的技能列表
> - `SkillTrait::dynamic_update_state_enabled()` + `Skill::dynamic_update_state_enabled()`：让外部能判断一个技能是否处于"短时持续生效"的运行时态
>
> 有了这两个接口，`tswn_wasm` 侧就可以在 `collect_states()` 里为每个玩家收集 `status_labels`（如"隐匿""聚气""狂暴""魅惑"等），并在前端渲染成彩色 pill 展示。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

`0.2.20` 把 win-rate 性能回退继续收窄了一大截，而本轮不需要继续碰性能路径或 parity 修复——那两条线已经稳定住了。

这次改动的直接触发点是：**WASM 前端 show 页面想要在玩家侧边栏显示实时状态标签**（比如"这个角色在隐匿""那个角色被魅惑了"）。

要在 WASM 层收集这些信息，需要 core 层提供两个先决能力：

1. **遍历玩家的技能** → `skill_storage()`
2. **判断技能是否激活了某个运行时态** → `dynamic_update_state_enabled()`

这两个接口本身非常简单，但它们涉及一个设计选择：要不要让 WASM 层直接依赖 core 的内部技能类型来做判断？

**选择是：不。**

WASM 层不直接 import `AccumulateSkill` / `HideSkill` / `ChargeSkill` 然后做 `downcast_ref`/`is`，而是通过 core 层提供的 trait 方法做统一查询。这样：

- WASM 层不需要知道具体有哪些技能实现了短时运行时态
- 未来新增技能时，只要实现了 `dynamic_update_state_enabled()`，WASM 层自动就能感知到
- `has_active_skill()` 辅助函数加了一层 `debug_skill_type_name().ends_with(suffix)` 的模糊匹配，用于区分具体是哪类技能在生效——这条路径只在 WASM 层使用，不影响 core 的 trait 边界

## 2. 主要改动

## 2.1 `Player::skill_storage()` — 暴露技能遍历入口

涉及文件：

- `crates\tswn_core\src\player\impl_runtime.rs`

新增：

```rust
#[inline]
pub fn skill_storage(&self) -> &crate::player::skill::store::SkillStorage { &self.skills }
```

这条方法的定位是**零开销暴露引用**——不做任何 clone / 锁 / 校验，直接返回内部 `self.skills` 的引用。

`SkillStorage` 的 `store` 字段是 `AHashMap<SkillId, Skill>`，外部拿到后可以做 `iter()` / `values()` 等遍历操作。

之所以需要这个方法，是因为之前 `SkillStorage` 是 `pub(super)` 可见性的——只在 `player` 模块内部可见，`tswn_wasm` 层无法直接访问。现在通过 `pub` 方法暴露出来，外部代码就可以用 `player.skill_storage().store.values()` 遍历技能了。

## 2.2 `dynamic_update_state_enabled()` — 技能运行时态标识

涉及文件：

- `crates\tswn_core\src\player\skill.rs`
- `crates\tswn_core\src\player\skill\act\accumulate.rs`
- `crates\tswn_core\src\player\skill\skl\hide.rs`

### `SkillTrait` 层

```rust
/// 仅供读取短时 update_state 运行时态使用。
/// 默认关闭，只有少数带"一段时间内持续生效"的技能会在激活时返回 true。
fn dynamic_update_state_enabled(&self) -> bool { false }
```

这是一个 trait 层的默认方法，返回 `false`。只有确实实现了短时运行时态的技能才需要 override 它。

### `Skill` 层

```rust
pub fn dynamic_update_state_enabled(&self) -> bool {
    self.level > 0 && self.skill_type.dynamic_update_state_enabled()
}
```

`Skill` 结构体在转发时额外加了一个 `level > 0` 守卫——也就是说，即使某个技能类型覆盖了 `dynamic_update_state_enabled()` 返回 `true`，但只要玩家还没学习这个技能（`level == 0`），仍然返回 `false`。

这个守卫很重要，因为 `SkillStorage` 的 `store` 中包含了所有注册过的技能，包括 `level == 0` 的未学习技能。如果不加这个守卫，外部遍历时会把没学的技能也算进去。

### 具体实现

**`AccumulateSkill`**（`act/accumulate.rs`）：

```rust
fn dynamic_update_state_enabled(&self) -> bool { self.on_update_state.is_some() }
```

**`HideSkill`**（`skl/hide.rs`）：

```rust
fn dynamic_update_state_enabled(&self) -> bool { self.on_update_state.is_some() }
```

两者都判断 `self.on_update_state.is_some()`——也就是说，只有当技能确实配置了"短时持续生效"的 `on_update_state` 效果时，才返回 `true`。

为什么这两个技能需要这个判断？因为它们的运行时态不是"一激活就一直持续到结束"，而是"只有在 `on_update_state` 配置了具体效果时才进入短时持续状态"。这个"短时持续"的语义在 JS 里是通过 `meta("update_state")` / `meta("charge")` 来标识的，而在 Rust 侧，`charge_runtime_active()` 已经覆盖了 `ChargeSkill` 的蓄力态，`dynamic_update_state_enabled()` 则覆盖了 `AccumulateSkill` 和 `HideSkill` 的 `update_state` 态。

## 3. WASM 层的配合改动

虽然这轮 `tswn_core` 只新增了两个接口，但 `tswn_wasm` 层基于这两个接口构建了完整的**状态标签 + 逐段渲染**系统：

### 3.1 状态标签收集

在 `tswn_wasm::fight::collect_states()` 中，为每个玩家收集实时状态标签：

- **技能运行时态**（通过 `has_active_skill()` + `*_enabled()` 查询）：
  - `聚气`：`AccumulateSkill.dynamic_update_state_enabled()`
  - `蓄力`：`ChargeSkill.charge_runtime_active()`
  - `隐匿`：`HideSkill.dynamic_update_state_enabled()`
  - `潜行`：`AssassinateSkill.dynamic_pre_action_enabled()`

- **状态效果**（通过 `player.get_state::<T>()` 查询，无需 core 新增接口）：
  - 正面：`狂暴`(BerserkState)、`疾走`(HasteState)、`铁壁`(IronState)、`守护`(ProtectState)
  - 负面：`魅惑`(CharmState)、`诅咒`(CurseState)、`冰冻`(IceState)、`中毒`(PoisonState)、`迟缓`(SlowState)、`垂死`(UpgradeState)
  - `冰冻` 同时覆盖 `IceState` 与 `status.frozen`，去重

### 3.2 辅助函数

- `push_status_label()`：去重插入，避免同一标签被多次添加
- `has_active_skill()`：泛型辅助，遍历 `skill_storage().store` 按 `debug_skill_type_name` 后缀匹配 + 自定义 filter

### 3.3 前端渲染

- `show-render.js`：`renderStatusPill()` / `renderPlayerStatusPills()` 渲染彩色 pill
- `show.css`：`.status-pill.positive`（绿色，正面状态）/ `.status-pill.negative`（红色，负面状态）
- `show.js`：`buildFrameRows()` 逐段渲染，按 chunk delay 细分推进

## 4. 没有引入的潜在改动

这轮 scope 非常集中，没有触碰以下方向：

1. **没有新增具体的技能运行时态实现**——`dynamic_update_state_enabled()` 只有默认实现，`AccumulateSkill` 和 `HideSkill` 是对已有 `on_update_state` 逻辑的封装，不引入新行为。
2. **没有修改已有技能的运行逻辑**——所有改动都是新增 API / 新增字段 / 新增 is_enabled 查询，不改变已有战斗流程。
3. **没有重构 SkillStorage 的可见性**——`pub(super)` 不改，只是新增一个 `pub` 方法做受控暴露。
