# tswn_core 0.2.11 更新说明（prepare/raw 一致性补测、CAPI 调用约定补充与 core/capi/py lint 清理）

> `0.2.11` 这一版没有继续扩展新的 CLI 子命令，而是主要补三类“工程正确性”工作：
>
> 1. 把 `PreparedRunner` 与普通 `raw -> Runner` 路径的等价性，补成更系统的多模式测试；
> 2. 补充一条更直接的无 seed 一致性验证，并把 CAPI / C++ 调用中容易踩坑的 seed 语义和 Windows 编译方式整理成文档；
> 3. 顺手把 `tswn_core`、`tswn_capi`、`tswn_py`（不含 `tswn_ds3`）这一侧的格式化与 clippy 问题清掉，避免后续继续迭代时被 lint 噪音干扰。
>
> 简版条目请同步维护在：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

本轮版本的目标包括：

- 明确验证 `PreparedRunner` 与普通 `raw` 构造路径，在同样输入、同样 seed 下是否真正等价；
- 避免只凭单个 case 判断 prepare 逻辑“看起来没问题”，而是按多种模式批量比对；
- 再补一条“无 seed”场景的直接一致性测试，确认 `raw` 不给 seed 与 `prepared` 不给 seed 时不会分叉；
- 把 CAPI / C++ 调用中本次实际踩到的 prepared seed 语义、Windows 下动态/静态链接方式等经验整理成文档，减少后续重复踩坑；
- 清理 `tswn_core` / `tswn_capi` / `tswn_py` 里的 clippy 报警和格式问题，确保后续新增改动时更容易识别真正的新问题；
- 为 `tswn_capi` 的 `unsafe extern "C"` 导出补充更完整的中文 `# Safety` 说明，降低 FFI 使用误解风险。

## 2. 主要改动

## 2.1 `PreparedRunner` 与普通 raw 构造路径的一致性补测

涉及文件：

- `crates\tswn_core\src\engine\test\runner\simple.rs`

### 本次改动

这一轮新增了一组更系统的测试，不再只验证单个输入，而是直接在测试里模拟多种 case miner 常见模式，对比：

- `Runner::new_from_namerena_raw(raw)`
- `Runner::prepare_groups_with_eval_rq(groups, rq)` + `Runner::new_from_prepared_with_seed(prepared, seed)`

在相同输入、相同 seed 下是否一致。

新增测试覆盖以下模式：

- `1v1`
- `2v2`
- `3v3v3`
- `ffa_4`
- `ffa_6`
- `ffa_8`

每种模式固定生成 `10` 个 deterministic case，总计 `60` 个 case。

每个 case 不只是比较“最后谁赢了”，而是同时比较：

- `input_groups`
- `battle score`
- `winner`
- `full replay trace`

### 效果

这组测试的意义在于，它回答的是一个比“跑起来像没问题”更严格的问题：

- `PreparedRunner` 是否真的只是把“与 seed 无关的重型初始化”提前缓存，
- 而不会在排序、构造顺序、初始状态、随机流消费等细节上与普通 raw 路径产生偏差。

实测结果表明，在上述 `60` 个 case 上：

- prepare 路径与 raw 路径的 `input_groups` 一致；
- `winner` 一致；
- `battle score` 一致；
- `full replay trace` 也一致。

这说明至少在当前补测覆盖范围内，没有发现“prepare 少做一步 sort”或“prepare/raw 初始化分叉”之类的问题。

## 2.2 补充无 seed 场景下的 raw / prepared 一致性测试

涉及文件：

- `crates\tswn_core\src\engine\test\mod.rs`
- `crates\tswn_core\src\engine\test\runner\mod.rs`

### 本次改动

除了前面那组“多模式 + 多 case + 同 seed”的一致性测试之外，这一版又额外补了一条更直接的单元测试，用来回答一个很具体的问题：

- `raw` 路径不传 seed
- `PreparedRunner` 路径也不传 seed

二者是否一致。

这条测试的做法是：

1. 使用同一份不带 seed 的 raw 输入构造 `Runner::new_from_namerena_raw(...)`；
2. 对同一份输入执行 `split_namerena_into_groups(...)`，并显式断言：
   - 解析出的 `seed` 列表为空；
3. 使用同一份 groups 构造 `PreparedRunner`；
4. 再通过 `Runner::new_from_prepared_with_seed(&prepared, &[])` 构造“无 seed”的 prepared runner；
5. 比较两边的：
   - `winner`
   - battle score
   - 回合数
   - replay trace

为了让这条测试能复用既有 runner 测试的辅助逻辑，本次还顺手把测试模块中的少量辅助函数改成了可被同级测试模块复用的公开函数。

### 效果

这条测试通过后，可以确认一件更细粒度、也更贴近实际调用约定的事情：

- `raw` 不给 seed
- `prepared` 不给 seed

二者是一致的。

也就是说，这一轮实际排查后，真正需要额外注意的不是“prepared 无 seed 会不会和 raw 无 seed 不一样”，而是：

- `PreparedRunner` 在 **有 seed** 场景下，传入的 seed 必须保持与 raw 文本一致的完整 `seed:...` 行格式。

## 2.3 `tswn_core` 侧 clippy 低风险问题清理

涉及文件：

- `crates\tswn_core\src\player\skill\act\summon.rs`
- `crates\tswn_core\src\player\skill.rs`
- `crates\tswn_core\src\player\impl_runtime.rs`
- `crates\tswn_core\src\player\skill\store.rs`
- `crates\tswn_core\src\player\state.rs`
- `crates\tswn_core\src\player\test\skills.rs`

### 本次改动

这一轮顺手处理了一批语义明确、低风险的 clippy 问题，包括：

- `map_entry`
- `if_same_then_else`
- `collapsible_if`
- `explicit_auto_deref`
- `excessive_precision`

对于两个参数数量较多、但当前接口形态本身较稳定的 helper，则采用局部：

- `#[allow(clippy::too_many_arguments)]`

而不是为了消除 lint 强行重构整条调用链。

### 效果

这部分改动本身不改变对外功能语义，但有两个直接收益：

1. 后续继续开发时，clippy 输出更干净，不容易被历史噪音淹没；
2. 一些原本“虽然能跑但写法不够收敛”的实现细节，被统一整理到更符合当前代码风格的状态。

---

## 2.4 `tswn_capi` 的 `unsafe` 导出补充中文 `# Safety` 文档，并清理原始指针访问写法

涉及文件：

- `crates\tswn_capi\src\lib.rs`

### 本次改动

本次对 `tswn_capi` 的导出层做了两类整理：

#### 1. 为 `unsafe extern "C"` 接口补充中文 `# Safety`

包括但不限于：

- 释放函数（如 `tswn_str_free` / `tswn_bytes_free` / `tswn_runner_free`）
- `Runner` / `PreparedRunner` 构造函数
- `updates` / `winner` / `input_group` 等查询函数
- icon / win_rate / group_win_rate 等高层 helper

这些文档主要明确：

- 哪些指针必须来自本库
- 哪些输出指针必须可写
- 哪些缓冲区需要满足容量要求
- 返回对象该由谁释放

#### 2. 调整原始指针上的集合访问写法

针对 `input_groups.get(...)`、`updates.get(...)` 这类逻辑，改为先取显式引用变量再调用方法，避免：

- `dangerous_implicit_autorefs`
- 与 `needless_borrow` 之间的冲突写法

### 效果

这部分改动的价值在于：

- 对 C/C++ / FFI 调用方来说，接口约束更明确；
- 对维护者来说，`unsafe` 边界的意图更清晰；
- 也顺手让 `tswn_capi` 这一层能通过更严格的 lint 检查。

## 2.5 `tswn_py` 侧 lint 与类型声明同步整理

涉及文件：

- `crates\tswn_py\src\lib.rs`
- `crates\tswn_py\tswn_py\_types_runner.pyi`

### 本次改动

Python 绑定这边主要顺手做了两类收尾：

- 清理 `redundant_closure` 一类的 clippy 报警；
- 修正 `Runner.player_input_group_index(player_id)` 在类型存根中的返回类型，使其与 Rust 实现保持一致。

当前该接口的语义是：

- 找到则返回输入 `group_index`
- 找不到则返回 `None`

因此类型存根应表达为：

- `int | None`

### 效果

这部分虽然不大，但能避免两个问题：

- Rust 实现和 Python 类型提示不一致；
- 后续在 Python 侧使用时，IDE / 类型检查器给出误导性的提示。

## 3. 影响 / 对齐点 / 兼容性

这一版对 `tswn_core` 本身没有新增新的 CLI 用户入口，也没有引入新的公开 breaking API；它更偏向“正确性与维护性收敛”。

直接可感知的影响主要有：

- `PreparedRunner` 与普通 raw 构造路径的一致性，现在有了更系统的自动化测试保障；
- `tswn_core` / `tswn_capi` / `tswn_py` 这三块在排除 `tswn_ds3` 后，格式化与 clippy 状态更干净；
- `tswn_capi` 的 FFI 安全约束文档更加清晰，外部调用方更容易知道哪些指针和缓冲区是合法用法；
- Python 存根与实际实现的对齐程度更高。

换句话说，`0.2.11` 的价值不在于“多了一个大功能入口”，而在于：

- 把 prepare 路径的正确性验证补扎实了；
- 把 CAPI / C++ 调用时最容易出错的 seed 语义与 Windows 编译方式补成了可复用文档；
- 把 core/capi/py 这几块的工程质量往前推了一步。

## 4. CAPI / C++ 调用补充说明

除了 core 侧的一致性测试，这一版也顺手整理了实际调用 `tswn_capi` 时的两个关键经验。

### 4.1 `PreparedRunner` 的 seed 必须传完整 `seed:...` 行

在 CAPI / C++ 调用里，本次实际排查确认：

- `raw` 路径如果要带 seed，本质上是在原始输入文本中追加一整行：
  - `seed:33554431@!`
- `PreparedRunner` 路径若要保持与 `raw` 一致，也应传入同样格式的完整字符串；
- 如果只传裸 seed 值，例如：
  - `33554431@!`
    则可能与 `raw` 路径出现结果偏差。

因此，本轮已经把这一点同步补充到：

- `crates\tswn_capi\examples\prepared_runner.c`
- `crates\tswn_capi\examples\README.md`

以避免后续调用方继续把 prepared 的 seed 当成“裸值”来传。

### 4.2 Windows 下 C++ 编译补充整理

本轮还把实际验证过的 Windows C++ / CAPI 编译经验整理成单独文档：

- `docs\howto\capi_cpp_windows.md`

这份文档补充了：

- `clang++` 动态链接 `tswn_capi.dll.lib` 的推荐命令；
- `clang++` 静态链接 `tswn_capi.lib` 的推荐命令；
- Windows 下静态链接时可能还需要额外补 `ntdll.lib`；
- 为什么推荐直接 `#include "tswn_capi.h"`；
- 为什么不推荐在源码里继续写 `#pragma comment(lib, ...)`；
- `g++` / MinGW 使用 `.dll.lib` 时可能遇到的格式兼容问题与处理建议。

这一部分不是 `tswn_core` 核心逻辑改动，但它直接来自这次真实的 CAPI 调用验证过程，因此也一并归档在 `0.2.11` 的更新说明中。

    - `1v1`
    - `2v2`
    - `3v3v3`
    - `ffa_4`
    - `ffa_6`
    - `ffa_8`
  - 每种模式 `10` 个 deterministic case，共 `60` 个 case
  - 比较：
    - `input_groups`
    - `battle score`
    - `winner`
    - `full replay trace`
- `cargo test -p tswn_core no_seed_runner_and_prepared_runner_match -- --nocapture`
  - 直接验证：
    - `raw` 不给 seed
    - `prepared` 不给 seed
  - 比较：
    - `winner`
    - battle score
    - round count
    - replay trace

说明：

- 上述格式化与 clippy 检查范围明确不包含 `tswn_ds3`；
- 最终 `tswn_core` / `tswn_capi` / `tswn_py` 三个 crate 在该检查范围内已经收敛到无 warning 通过。
