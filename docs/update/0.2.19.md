# tswn_core 0.2.19 更新说明（收口 3v3v3 大样本 diff，并补跑当前性能口径）

> `0.2.19` 这一版的核心目标只有一个：继续把当前分支上这批多人对局分叉收口，直到 `track_case_miner` 在 `tests\sqp5900.txt` 的 `3v3v3 + maxcase=4000` 口径下，真正回到 `diff_failures = 0`。
>
> 这轮最终落地的关键不在单一技能，而是几条容易互相影响的运行时语义同时收紧：
>
> - revive / spawn / death 的落地顺序
> - `Hide` 在同一 action 后半段看到的 alive 视图
> - `Assassinate` / `Hide` 的 pre_action 累积语义
> - summon / merge / combat minion 的固定槽位与继承规则
>
> 另外，按 `docs\performance.md` 的口径补跑了一轮当前 release benchmark；结果一致性已经收口，但性能相较 `0.2.14` 基线整体偏慢，结论也一并记在本文。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

这一轮改动的出发点非常直接：

- 近期 `sby_test.md` / `track_case_miner.py` 路径下的多人回归已经压到只剩少量 3v3v3 diff_case；
- 这些 case 不是“引擎整体乱掉”，而是若干条相邻运行时语义各差 1 byte / 1 个 hook / 1 个可见实体，最终在后续 skill 选择、rapid 轮转、背刺/隐匿判定里被放大；
- 如果不把这些语义一次性钉住，后面再继续修 `skill` 区域时，很容易在旧 case 清零后又引入新 diff。

因此，`0.2.19` 主要处理五件事：

1. 明确 revive / spawn / death 的 sync 顺序；
2. 修正 `Hide` 在 post_damage / post_action 链里的 same-action alive 可见性；
3. 对齐 `Assassinate` / `Hide` 的 pre_action 强制出手累积语义；
4. 把 summon / merge / combat minion 的固定槽位与衍生体属性继续对齐到 JS；
5. 用更细的回归测试把上述语义锁住，并补跑当前 release 性能口径。

## 2. 主要改动

## 2.1 revive / spawn / death 改成“先 revive，再 spawn，最后 remove”

涉及文件：

- `crates\tswn_core\src\engine\engine_core.rs`
- `crates\tswn_core\src\engine\storage.rs`
- `crates\tswn_core\src\engine\test\runner\mod.rs`

这轮把 runtime sync 的顺序重新收口成：

1. 显式 `queue_revival`
2. fallback 扫描出的“已经在 storage 里活着、但 world 侧还没挂回 alive”的成员
3. `pending_spawn`
4. `pending_remove` / death cleanup

这样做是为了和 JS 保持同一条时序：

- revive 会先把旧实体重新挂回当前 roster / alive 视图；
- 如果同一 tick 后面又有 spawn（例如尸体转丧尸、召唤物落地），spawn 应排在 revive 后；
- 如果同一 tick 后面又发生死亡移除，也必须保持“先 revive / spawn，再 remove”。

同时在 `Storage` 新增了：

- `pending_revival_ids_for_group(...)`

供技能层在“同一 action 后半段，但 world 还没 sync”这类窗口里查询“哪些旧成员已经复活，只是 alive_group 还没更新”。

对应补上的回归测试：

- `sync_runtime_entities_applies_revivals_before_spawns_in_round_order`

它直接锁住“revived player 应该先回队伍，再让 spawn 挂进末尾”的顺序。

## 2.2 `Hide` 的 same-action alive 视图继续对齐 JS

涉及文件：

- `crates\tswn_core\src\player\skill\skl\hide.rs`
- `crates\tswn_core\src\player\test\skills.rs`

`Hide` 这一轮的修正点有两个，而且两个方向都必须同时满足：

### 1. 要看到“刚复活 / 刚 addNew”的队友

JS 的 `Hide.aD()` 读取的是当前 action 后半段的 alive roster 视图。也就是说：

- owner 当前 action 内刚 `addNew` 出来的 pending spawn
- 同拍刚 `queue_revival` 的旧队友

在 poison / post_damage / post_action 这些后续 hook 来到 `Hide` 时，应该已经可见。

Rust 这边现在会把：

- `pending_spawn_ids_for_owner(...)`
- `pending_revival_ids_for_group(...)`

补进 `alive_candidates`，从而修掉此前 `f250` 那类“同拍 revive 后，Hide 少吃 1 byte”的问题。

### 2. 不能把“已经死了、只是 alive_group 还没来得及 sync 掉”的队友算进去

这也是本轮真正导致新回归的坑点：

- `alive_group` 在某些 post_damage 窗口里可能仍残留刚死的队友；
- 如果直接拿 snapshot 长度计数，就会让 `Hide` 平白多做一次 `r63()`；
- 这会立刻把 rapid / skill selection / 后续 action 的 RC4 全部推偏。

因此现在会先用 `player.alive()` 过滤一遍 snapshot，再追加 pending revival / pending spawn。

新增回归测试：

- `hide_counts_revived_ally_as_alive_before_sync`
- `hide_ignores_unsynced_alive_ally_without_pending_revival`
- `hide_ignores_dead_ally_left_in_alive_group_until_sync`

这三条组合起来，正好把“该看见的要看见、不该看见的不能误算进去”这条语义钉死。

## 2.3 `Assassinate` / `Hide` 的 pre_action 累积语义继续对齐

涉及文件：

- `crates\tswn_core\src\player\impl_runtime.rs`
- `crates\tswn_core\src\player\skill.rs`
- `crates\tswn_core\src\player\skill\store.rs`
- `crates\tswn_core\src\player\skill\act\assassinate.rs`
- `crates\tswn_core\src\player\skill\skl\hide.rs`
- `crates\tswn_core\src\player\test\skill_store.rs`

这轮给技能框架补了几条更接近 JS 的运行时能力：

- `pre_action_accumulate(...)`
- `dynamic_pre_action_enabled()`
- `manages_dynamic_pre_action()`
- `allows_empty_targets()`
- `uses_attack_aa_sampling()`

背后的目标，是把 JS 里“上一项 preAction 返回值如何累积 forced skill”的语义翻译得更像：

- `Assassinate` 在 pending target 仍活着时，应该无视 `prev`，直接把自己设成 forced；
- `Hide` 进入 pre_action 时只负责清自己的状态，不应错误保留旧 forced skill；
- 某些技能对外表现为“targets = [] / null”，但内部依然保留了真实目标，act 阶段应允许继续从内部 target 出手；
- `EnemyAlive` 的一部分技能虽然有自己的 `aa()`，但候选抽样仍复用 ActionSkill 的 `attack_aa` 路径，不能把候选提前压缩成紧凑列表，否则 RC4 会变轻。

这一组改动让：

- pending 背刺目标
- late-gained hide
- attack_aa 风格的 skill target sampling

在 Rust 里继续和 JS 对齐。

新增回归测试包括：

- `assassinate_pre_action_is_preserved_by_shield`
- `late_gained_hide_clears_assassinate_forced_skill`

## 2.4 summon / merge / combat minion 的固定槽位语义收紧

涉及文件：

- `crates\tswn_core\src\player\skill.rs`
- `crates\tswn_core\src\player\skill\store.rs`
- `crates\tswn_core\src\player\skill\skl\merge.rs`
- `crates\tswn_core\src\player\skill\act\summon.rs`
- `crates\tswn_core\src\player\skill\act\minion.rs`
- `crates\tswn_core\src\player\skill\act\shadow.rs`
- `crates\tswn_core\src\player\skill\skl\zombie.rs`
- `crates\tswn_core\src\player\test\minions.rs`

这一块的核心收口，是把：

- 固定槽位视图
- 当前主动技能扫描顺序

明确拆成两层。

`SkillStorage` 现在同时维护：

- `slot_skill`：对齐 JS `k1` 的固定技能槽位视图
- `skill`：对齐 JS `k4` / 当前 action 扫描顺序

这样可以覆盖两类过去很容易混掉的语义：

### 1. summon 的固定槽位不能跟着 action 顺序洗牌

JS summon 的关键是：

- 固定槽位 `k1` 仍是 `[fire, fire, explode]`
- 只是遍历顺序 / 当前主动对象顺序会变化

如果 Rust 把这两者混成同一份数组，后面一旦有人吞 summon，merge 就会按错误顺序继承等级。

### 2. merge 继承的是“固定槽位位置”，不是 skill key 排序结果

现在 `merge` 改成按 `slot_skill` 逐位继承等级，而不是再把技能按 key 排序后对齐。

配套新增 / 调整的回归测试包括：

- `summon_merge_uses_fixed_skill_slots`
- `merge_inherits_levels_by_fixed_slot_even_when_runtime_kind_differs`
- `merge_inherits_fengshen_shield_and_registers_pre_action`

### 3. combat minion 的 `name_factor` 统一归零

JS `Minion.bf()` 会把 shadow / summon / zombie 这类战斗衍生体的 `x/name_factor` 置成 `0`。

Rust 这边现在提炼了：

- `prepare_combat_minion(...)`

并统一在 shadow / summon / zombie 路径调用，避免衍生体还带着普通玩家的名字因子。

对应测试里也补上了：

- shadow / summon / zombie 衍生体的 `name_factor == 0.0`

## 3. 性能口径补跑

按 `docs\performance.md` 的口径，使用：

- `--release --features no_debug`
- `tswn-cli bench win-rate ... --perf`
- 单线程 / CLI 默认多线程
- `100k` 与 `1M`

补跑当前工作树，结果如下：

| 样本           | 单线程 `100k` | 多线程 `100k` | 单线程 `1M` | 多线程 `1M` |
| -------------- | ------------: | ------------: | ----------: | ----------: |
| `aaa` vs `bbb` |      `1.978s` |      `0.309s` |   `19.246s` |    `2.743s` |
| `复杂样本`     |      `2.895s` |      `0.394s` |   `28.692s` |    `3.900s` |

和 `docs\performance.md` 里当前记录的 `0.2.14` 基线相比，这轮 8 项都偏慢：

- `aaa`：单线程慢约 `6%~8%`，多线程慢约 `9%~11%`
- 复杂样本：单线程慢约 `3%~4%`，多线程慢约 `5%~12%`

最明显的两项是：

- `aaa` `1M` 多线程：`2.516s -> 2.743s`，慢约 `9.0%`
- 复杂样本 `1M` 多线程：`3.473s -> 3.900s`，慢约 `12.3%`

也就是说，`0.2.19` 目前可以明确确认的是：

- **结果一致性收口了**
- **多人 diff_case 清零了**
- **但性能还没有回到 `0.2.14` 的基线**

后面如果继续做性能轮，需要优先盯：

- `bench win-rate` 多线程路径
- 复杂样本的 `fight` 热路径
- 这轮新增的动态 pre_action / fixed slot / same-action visibility 相关代码段
