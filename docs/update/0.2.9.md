# tswn_core 0.2.9 更新说明（修复 Protect 防御链提前刷新疾走倍率）

> `0.2.9` 是一版很典型的“最后一处时机误差把整条分支带歪”的修复：表面上看是 `3v3v3` 中某个玩家多动了一次、背刺伤害对不上、默认攻击目标也错了，但根因其实是一条 `pre_defend` 清态路径把 `Haste` 的延迟倍率提前刷进了 `speed`。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

`0.2.8` 之后，固定大样本里只剩最后两个 `3v3v3` diff case。

这两个 case 的共性都不是“某个技能完全没实现”，而是：

- 某个玩家会在中段多拿到一次行动；
- 之后默认攻击目标、背刺伤害和 RC4 全部连锁漂移；
- 表面看像是 `Haste`、`Reraise`、`Protect`、`Assassinate`、`round_pos` 都有嫌疑。

最终排查结果表明，真正的问题点很窄：

- `ProtectState` 相关的 `pre_defend` 清态路径；
- 在这条路径里，Rust 还会无条件 `update_states()`；
- 从而把本该延迟到后续时机才生效的 `HasteState.faster` 提前刷进当前 `speed`。

---

## 2. 主要改动

## 2.1 `pre_defend` 清态只在必要时才重算属性

涉及文件：

- `crates\tswn_core\src\player\impl_runtime.rs`
- `crates\tswn_core\src\player\state.rs`
- `crates\tswn_core\src\player\skill\skl\protect.rs`

### 本次修正

- 给 `StateTrait` 增加 `clear_updates_status()`，默认返回 `true`；
- `PlayerStateStore` 增加 `tag_clear_updates_status()` 查询；
- `ProtectState` 明确标记为 `clear_updates_status() -> false`；
- `clear_state<T>()`、`pre_step`、`post_action`、普通 `pre_defend`、以及 `Protect` 专门分支中的 `pre_defend` 清态，都改为：
  - 只有被清掉的状态里确实存在“会影响属性”的项时，才执行 `update_states()`。

### 为什么这能修正最后的分支漂移

JS 里 `Haste + charge` 的语义是：

- 先让基础 `z=2` 的 haste 立即生效；
- 再把 `z/Q` 改成 `4/5`；
- 但不会在同一时刻再次 `F()`，所以强化倍率不会立刻进入当前 `speed`。

Rust 之前的问题是：

- `Protect` 相关的 `pre_defend` 清态在一次普通受击后又额外 `update_states()`；
- 这会把 `HasteState.faster=4` 提前刷进 `speed`；
- 导致本不该过线的 actor 过线、多拿一次行动；
- 后面就会进一步表现成：
  - 默认攻击目标不同；
  - 背刺的 `getAt` 三次取样不同；
  - RC4 全链路偏移。

把这条误刷路径关掉后，整条分支自然回到 JS 顺序。

---

## 2.2 `Reraise` 语义继续保持与 JS 一致

涉及文件：

- `crates\tswn_core\src\player\skill\skl\reraise.rs`

### 本次说明

- `Reraise` 仍然不恢复 `queue_revival()`；
- 继续保持“只在当前死亡链里直接回 HP，不额外排一次 revival/sync”的语义；
- 在代码里补了中文注释，说明这条路径已经经过全量 `1w2 case` 验证，不要轻易改回去。

### 效果

这不是本轮最终命中的根因，但它是一个已经验证过的 JS 语义点：

- 保留当前实现不会引入回归；
- 也避免后续排查时有人把它误当成“肯定要补一次 revival sync”。

## 2.3 调试探针与 md5.js 注释继续保留

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli.rs`
- `crates\tswn_core\src\engine\engine_core.rs`
- `crates\tswn_core\src\engine\world_state.rs`
- `crates\tswn_core\src\player\impl_attr.rs`
- `crates\tswn_core\src\player\skill.rs`
- `crates\tswn_core\src\player\skill\act\assassinate.rs`
- `crates\tswn_core\src\player\skill\act\charge.rs`
- `crates\tswn_core\src\player\skill\act\haste.rs`
- `crates\tswn_core\src\player\skill\act\iron.rs`
- `crates\tswn_core\src\player\skill\store.rs`
- `fast-namerena\branch\latest\md5.js`
- `fast-namerena\branch\latest\out_md5.ts`

### 本次保留内容

- `raw update`、`tick_end`、`remove_player`、`update_states` 等 Rust 探针；
- `haste / iron / assassinate / default attack` 等局部窗口探针；
- `md5.js` 中关于 `Reraise`、`Haste`、默认攻击等关键片段的中文注释。

### 价值

这一轮之所以能把问题从“最后两个 3v3v3 diff”缩到一条 `pre_defend -> clear_tag -> update_states` 路径，靠的就是这些定向探针和参考侧注释。

因此 `0.2.9` 继续把它们保留下来，供后续继续对账时复用。

---

## 3. 影响 / 对齐点 / 兼容性

`0.2.9` 没有新的外部 API breaking change，但行为层面是一个重要收敛点：

- `Protect` 相关受击链不会再提前刷新 `Haste` 强化倍率；
- 由此带来的行动阈值、默认攻击目标、背刺数值与 RC4 顺序漂移都会一起消失；
- 固定 SBY 大样本中，Rust 与 TS 基准已经对齐到 `diff_failures = 0`。

如果你在做 raw log / case miner 对账，这一版就是“最后一处大样本分歧消失”的版本；
如果你只是调用 crate 的上层接口，这版主要是行为精修，不涉及接口迁移。
