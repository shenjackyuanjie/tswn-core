# tswn_core 0.2.0 详细更新说明

> 本文是 `0.2.0` 的工程说明版，不只列“改了什么”，还解释“为什么这样改、怎么落地、效果如何、后续怎么演进”。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 这次版本要解决的核心问题

这次改动围绕三个长期痛点：

1. **状态系统过度依赖 `Any/downcast`**
   - 每个状态要重复写 `as_any/as_any_mut` 样板；
   - 代码可读性差，扩展时心智负担大；
   - 未来 DLL/插件边界上，`TypeId/Any` 语义不够“协议化”。

2. **技能/Boss 扩展仍偏“硬编码分发”**
   - 新增能力常常要改核心 `match`；
   - 对外扩展入口不稳定，插件化不友好。

3. **`--win_rate` 热路径性能不足**
   - 早期基线下 `10000` 场对战耗时约 `0.422s`（release + no_debug，直接运行二进制）；
   - 初始化、日志构建、循环内分配和串行执行都存在优化空间。

---

## 2. 设计思路（为什么这样做）

本次思路是：**先把“扩展能力边界”做对，再把“热路径成本”做低**。

- 对扩展性：把“核心 hardcode 分发”改成“注册中心 + 稳定入口”。
- 对复杂度：状态系统从 `Any` 过渡到“显式标签 + 类型校验 + 局部 unsafe”。
- 对性能：对 bench 路径做“只做必须做的事”：
  - 不需要日志帧时就不构建；
  - 能复用的输入结构就复用；
  - 能并行就并行，并且做动态负载均衡；
  - 小集合用 `SmallVec` 减少堆分配。

---

## 3. 具体改动（原因 / 实现 / 效果）

## 3.1 状态系统去 `Any`（Breaking Change）

### 原因

- `Any/downcast` + `as_any/as_any_mut` 在状态类型多时会导致大量重复样板；
- 对插件边界不够直观，不利于定义稳定协议。

### 实现

主要在 `crates\tswn_core\src\player\state.rs`：

- 引入 `StateTag = &'static str`，默认由 `type_name::<T>()` 提供稳定标签；
- `StateTrait` 增加：
  - `state_tag()`：稳定可读标识；
  - `state_type_id()`：内部类型校验；
- `PlayerStateStore` 由 `StateTag -> Box<dyn StateTrait>` 持有；
- `get/get_mut` 使用 `state_type_id` 校验后再做指针转换（`cast_ref/cast_mut`），并补充了 `SAFETY` 注释；
- 删除状态实现里原有的 `as_any/as_any_mut` 样板（涉及多个 act/skl/boss 状态实现文件）。

### 效果

- 状态实现更轻，样板显著减少；
- 运行时行为不变，类型恢复逻辑集中管理；
- 为后续跨边界状态协议（DLL/hook）留出更清晰的标识层。

---

## 3.2 技能系统注册中心化 + 快路径优化

### 原因

- 旧的技能实例化依赖固定分发逻辑，扩展需要改核心；
- 在高频对战下，技能构造路径上的全局锁访问会产生额外开销。

### 实现

主要在 `crates\tswn_core\src\player\skill.rs`：

- 新增 `SkillRegistry`（`OnceLock<RwLock<...>>`）；
- 新增公开注册入口 `register_skill_factory(id, factory)`；
- 内置技能工厂改为 `BUILTIN_SKILL_FACTORIES` 常量数组；
- 新增 `SKILL_REGISTRY_DIRTY: AtomicBool`：
  - 未发生外部注册时，`create_skill_from_registry` 走**无锁内置快路径**；
  - 发生覆盖注册后再走 registry 读锁路径；
- `SkillExt` 不再要求 `Any`。

### 效果

- 扩展从“改核心代码”变为“注册工厂”；
- 默认内置运行时不为扩展能力付出不必要锁成本；
- 和 3.1 一起完成“去 Any + 可扩展”的基础能力。

---

## 3.3 Boss 系统注册中心化

### 原因

- Boss 的初始化/行动/免疫策略此前分散在硬编码路径中，新增 Boss 成本高；
- 未来插件式 Boss 注入需要统一扩展点。

### 实现

主要在 `crates\tswn_core\src\player\boss\mod.rs`：

- 新增 `BossHandler`（`init_state` / `action_prob_count` / `immune_threshold` / `default_action`）；
- 新增 `BossRegistry`，内置注册 `covid/lazy/saitama`，其余走 fallback；
- 新增 `register_boss_handler(name, handler)`；
- `init_boss_state`、`boss_action_prob_count`、`boss_immune_threshold`、`boss_default_action` 全改为 registry 驱动。

### 效果

- 新 Boss 扩展可以通过注册完成，不再改核心分发；
- 行为入口统一，后续 DLL 注入更自然。

---

## 3.4 Hook 管线增加动态扩展通道

### 原因

- 函数指针 hook 在内置场景性能好，但对动态加载/跨模块扩展不够灵活。

### 实现

主要在：

- `crates\tswn_core\src\engine\hooks.rs`
- `crates\tswn_core\src\engine\engine_core.rs`

新增：

- `ActorHookDyn` trait；
- `HookPipeline` 的 `*_dyn` 存储与执行通道；
- `EngineCore` 对外 `register_*_hook_dyn` API（pre/post action + pre/post damage）。

### 效果

- 保留函数指针快路径；
- 同时具备 trait object 动态扩展能力；
- 达到“性能与可扩展并存”的设计目标。

---

## 3.5 `RunUpdates` 无帧采集高速路径

### 原因

`run_to_completion` 仅关心“本轮是否有事件/是否结束”，并不需要完整事件文本；  
继续构建 `RunUpdate` 会产生大量无收益开销（字符串、对象、push）。

### 实现

主要在：

- `crates\tswn_core\src\engine\update.rs`
- `crates\tswn_core\src\engine\runners.rs`
- `crates\tswn_core\src\engine\tick.rs`

改动点：

- `RunUpdates` 新增：
  - `capture_updates: bool`
  - `has_activity: bool`
  - `new_no_capture()`、`reset()`、`had_updates()`、`emit(...)`
- `add()` 在 no-capture 下只标记活动，不保存帧；
- `run_to_completion()` 改用 `RunUpdates::new_no_capture()`；
- tick 轮询判断改为 `had_updates()`。

### 效果

- bench/胜率路径大幅减少无意义对象构建；
- 不影响普通可视化对战路径（`RunUpdates::new()` 仍保留完整帧）。

---

## 3.6 `--win_rate` 端到端热路径优化

### 原因

CLI 的胜率测试本身是纯批处理任务，天然适合并行且可复用输入解析结果；  
原始实现在循环里有较多重复工作（构造原始字符串、重新解析分组、串行执行）。

### 实现

主要在：

- `crates\tswn_core\src\bin\tswn_cli.rs`
- `crates\tswn_core\src\engine\runners.rs`

关键策略：

1. **分组解析复用**
   - 只解析一次 raw；
   - 新增 `Runner::new_from_groups_with_seed(players, seed)` 供高频场景复用。

2. **并行执行**
   - `run_bench_winrate` 改为多线程；
   - 默认 worker 数为 `available_parallelism * 1.25`（向上取整）；
   - 支持 `TSWN_WINRATE_WORKERS` 环境变量覆盖。

3. **动态负载均衡**
   - 使用 `AtomicUsize` 分发任务索引（work-stealing 风格）；
   - 比静态切块更能应对单局耗时波动。

4. **循环内低分配**
   - seed 文本复用 `String` + `write!`；
   - 胜者判定用 team0 id 区间判断，避免额外集合比较。

### 效果

- `--win_rate` 成为纯计算导向路径，吞吐显著提升；
- 并发配置可在不同机器上调优；
- 保持原有输出语义（胜率/总场次）。

---

## 3.7 目标选择热路径小对象优化（`SmallVec`）

### 原因

攻击与技能目标选择多数是“小集合”（2~4 个目标），  
使用 `Vec` 会带来频繁小堆分配。

### 实现

主要在 `crates\tswn_core\src\player\impl_runtime.rs`：

- `selected`、`scored` 等中间集合改为 `SmallVec`；
- 默认攻击、强制攻击、技能选目标等路径统一收敛到低分配实现；
- 配合 `updates.emit(...)` 避免 no-capture 场景无效构建 `RunUpdate`。

### 效果

- 高频循环下减少堆分配压力；
- 对 CPU cache 友好，和并行优化叠加收益明显。

---

## 3.8 Release 编译配置优化

### 原因

高频短函数、跨模块调用较多，适合更激进的链接期优化。

### 实现

`Cargo.toml`（workspace）新增：

```toml
[profile.release]
codegen-units = 1
lto = "fat"
```

### 效果

- 更强的跨单元优化机会；
- 编译时间有所增加，但运行时吞吐提升（尤其在 benchmark 场景）。

## 4. 性能验证与结果

## 4.1 验证方法

建议用两步法，避免把 Cargo 包装层开销混入业务测量：

```powershell
cargo build --release --features no_debug --bin tswn-cli
.\target\release\tswn-cli.exe --win_rate aaa bbb 10000
```

## 4.2 结果（同机对比）

- 基线（改造前，直接二进制）：
  - `--win_rate aaa bbb 10000` 平均约 **0.422s**
- 改造后（本次）：
  - 稳态多次平均约 **0.074~0.076s**
  - 典型单次可见约 **0.064s**

> 说明：结果会随 CPU 频率、系统负载、线程调度有波动。

## 4.3 目标达成情况

- 用户期望：尽量逼近 `0.05s`
- 当前达成：**接近目标但未稳定进入 0.05s 内**，已显著下降（约 5~6 倍提升，视口径而定）。

## 5. Breaking Change 清单与迁移建议

## 5.1 Breaking Change

1. 状态 trait 不再使用 `as_any/as_any_mut`。
2. `SkillExt` 去掉 `Any` 约束。
3. `RunUpdates` 新增采集策略（对依赖其内部行为的调用方需确认预期）。

## 5.2 迁移建议

- 自定义状态实现：
  - 删除 `as_any/as_any_mut`；
  - 保留/实现 `clone_box`；
  - 如需自定义标签可覆盖 `state_tag()`。
- 自定义技能/Boss：
  - 改为在初始化阶段调用 `register_skill_factory` / `register_boss_handler`。
- 若要做动态 hook：
  - 优先使用 `register_*_hook_dyn`，并在外层控制生命周期与线程安全。
