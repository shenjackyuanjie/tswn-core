# tswn_core 0.3.10 更新说明

`0.3.10` 是一次以 **CLI / 包装层 API 对齐**、**DIY 召唤物导出补全** 和 **OpenHarmony/OHOS 构建支持** 为主的版本。相比 `0.3.9`，这一版继续收口 `tswn-cli`、Python、WASM、C-API 之间的高层 helper 语义，同时补齐服务端 replay 相关接口，并把鸿蒙 / OpenHarmony 的 `tswn-cli` 交叉编译流程接入聚合打包。

本版本的核心变化可以分成三条线：第一，`tswn_core::cli_api` 成为 CLI、Python、WASM、C-API 共用的高层逻辑层，减少多端重复实现带来的语义漂移；第二，DIY / OL 的衍生体导出继续补全，尤其是 shadow、summon、zombie 与嵌套召唤物模板；第三，新增 OHOS native SDK 环境脚本和 `build_all.py --include-ohos-cli`，可以在完整发布包中产出未签名的鸿蒙 `tswn-cli` ELF 二进制。

## 版本定位

- 当前版本：`tswn_core 0.3.10`
- 上一版本：`tswn_core 0.3.9`
- 重点方向：
  - `tswn_core::cli_api` 共享高层 helper；
  - Python / WASM / C-API 与 CLI 语义对齐；
  - 服务端 replay / live helper 补齐；
  - DIY summon / shadow / zombie 模板导出增强；
  - `bench win-rate` 输入与输出修复；
  - namer-pf 分项、精度和 JS 边界继续对齐；
  - OpenHarmony/OHOS 交叉编译与聚合打包支持。

## CLI 与共享 API

### 新增 `tswn_core::cli_api`

本版本新增 `tswn_core::cli_api` 模块，用于集中承载原先分散在 `tswn-cli`、`tswn_py`、`tswn_wasm` 和 `tswn_capi` 中的高层辅助逻辑。

新增或统一的能力包括：

- `win_rate_summary`
- `team_win_rate_summary`
- `group_win_rate_summary`
- `score`
- `namer_pf`
- `batch_rate`
- `pair_rate`
- `to_diy`
- `to_diy_batch`
- `icon_info`
- `parse_group_lines`

这些接口的目标不是替代底层 `Runner` / `PreparedRunner`，而是为“和 CLI 输出语义一致”的包装层提供统一入口。之后 Python、WASM、C-API 不需要各自维护一份 batch-rate、pair-rate、to-diy、icon info 等逻辑，减少语言边界之间的行为分叉。

### CLI 模块拆分

`tswn_cli` 内部做了一轮结构化拆分：

- `args/`：命令行参数、输入源和解析后参数；
- `bench/`：batch / score / winrate / common / output；
- `fight/`：driver / trace / raw_bench；
- `icon`、`to_diy` 等命令继续保持独立职责。

这次拆分不改变命令的主要语义，主要是降低后续维护 CLI 的耦合度，并方便共享逻辑迁移到 `cli_api`。

## CLI 功能与修复

### `namer-pf` 分项和精度

`namer-pf` 新增 `--mode` 参数，支持只计算指定评分项：

- `pp`
- `pd`
- `qp`
- `qd`

可以重复传入，也可以逗号分隔。不传时仍默认运行四项。

同时新增 `--precision <N>`，控制输出小数位。默认保持整数输出；显式设置后可以输出真实小数分数，避免只在整数后补零。

### `bench win-rate` 输入修复

`bench win-rate` 的输入方式调整为更明确的 `-r/--raw` 或 `-f/--file`：

- 两队之间用空行 / 换行分隔；
- 队内默认用 `+` 分隔；
- 新增 `--double-plus`，可改用 `++` 作为队内分隔符，避免名字里的 `+diy[...]` 或 `+ol:{...}` 被错误切开。

同时修复了 `bench win-rate` 未对 `team1` / `team2` 调用 `decode_raw` 的问题。现在命令行传入的 `\n` 能像 `bench group-win-rate` 一样被正确解析成换行。

### `bench win-rate --keep-rq` seed 调度修复

修复 `bench win-rate --keep-rq` 会同时切换 seed 调度的问题。现在 `--keep-rq` 只影响 `rq`，胜率模拟仍使用 JS `ProfileWinChance` 的 seed 口径：

- 首场无 seed；
- 后续从 `seed:(33554431 + i)@!` 开始递增。

这避免了 rq 与 seed 调度两个维度混在一起，方便做对照排查。

### 终端输出改进

普通 `bench win-rate` 对战胜率测试现在会先打印对战双方或队伍信息，便于直接确认当前 matchup，减少大批量脚本中输错输入却不容易发现的问题。

## DIY / OL 导出

### 衍生体模板导出补全

`to-diy --minions` 继续增强 shadow、summon、zombie 模板导出，使 DIY / OL 输出能更完整地还原战斗衍生体属性和技能。

本轮重点包括：

- shadow / summon / zombie 模板可携带更完整的属性与技能配置；
- 支持嵌套 DIY 召唤物模板；
- summon 模板中的子 summon 配置可以继续导出和回灌；
- `ol` overlay 中的召唤物模板与主玩家属性分离更清晰。

这让 `to-diy --minions` 更适合作为复杂 DIY 名字的迁移和复测工具，而不只是导出主玩家。

### summon 技能格式收紧

summon 模板技能格式从旧的数组 / `skill_order` 组合，收口为普通 JSON object：

- `sklfire1`
- `sklfire2`
- `sklexplode`

字段顺序即行动顺序。旧的 `sklfire`、`fire1`、`explode` 等别名不再兼容。`to-diy --minions` 会省略 0 熟练度使魔技能，但内部固定槽位仍保留；吞噬使魔时继续按固定槽位继承等级。

这样做的目标是减少重复状态和重复技能名带来的歧义，尤其是 summon 被吞噬、继承或嵌套召唤时的槽位映射问题。

### `name_factor_enabled` 解析修复

修复 `ol:{...}` overlay 中 `name_factor_enabled` 字段未被解析的问题。JSON 格式的 OL overlay 现在能正确识别该开关，用于控制 `name_factor` 是否参与属性缩放。

## 行为对齐修复

### namer-pf JS 边界继续修复

本版本继续修复 namer-pf 小分差对账中的 JS 行为边界：

- `Hide` 对被 charm 后的同队 alive 视图，会同时纳入 pending revival 与同队 pending spawn，避免隐匿检定少消耗随机字节；
- `Protect` 在 merge / devour 后改为读取保护者当前 `ProtectSkill` 等级，使 `r127() < level` 判定与 JS 产物保持一致；
- 清除 `BerserkState` 时不再误刷新 pending 正面状态，避免蓄力中的 `HasteState.faster` 提前写回 speed；
- `main_round()` 在任意可见 update 产生后即停止继续推进，避免尾部状态解除、毒性发作等事件和下一名主体行动合并到同一批更新里；
- `prepared_win_rate` 在非 JS profile rq 下不再错误套用 profile seed 计划。

这些修复仍以减少 Rust 和 md5.js / Bun 侧 namer-pf 分叉为目标。

### 胜利队伍映射

`WorldState` / `Runner` 现在记录内部战斗队伍到原始输入队伍的映射，并提供：

- `winner_team_index()`
- `winner_team_indices()`

包装层可以直接获得权威胜利输入队伍，不必再从胜者 player id 反推。这样可以避免分身、召唤物、幻影、丧尸等运行时实体影响胜者队伍判断。

## Python / WASM / C-API

### Python 高层 helper 补齐

`tswn_py` 新增一批顶层 helper，与 `tswn-cli` 的常用功能对齐：

- 胜率摘要：`win_rate_summary()`、`team_win_rate_summary()`、`group_win_rate_summary()`
- 评分与 namer-pf：`score()`、`namer_pf()`
- 批量胜率与配队：`batch_rate()`、`pair_rate()`
- DIY / OL 导出：`to_diy()`、`to_diy_batch()`
- 图标元信息与列表解析：`icon_info()`、`parse_group_lines()`

同时新增结构化返回类型和类型存根，方便 Python 调用方直接消费结果。

### Python replay / live helper

Python 绑定新增服务端直播 / 回放相关接口：

- `Runner.winner_team_index()` / `winner_team_indices()`
- `Runner.snapshot_players()`
- `RunUpdate.to_dict(rendered=True)`
- `Runner.build_replay(limit=None)`
- `compute_show_timeline(updates, player_count, scale=True)`

这些接口使用每个引擎 tick 前后的真实状态快照，服务端不需要再根据事件文本自行模拟逐条状态变化。

### WASM 高层导出

## OpenHarmony / OHOS / 鸿蒙

### 交叉编译环境脚本

本版本新增 OpenHarmony / OHOS 交叉编译辅助脚本和文档。仓库不提交本机 SDK 绝对路径，而是通过环境变量指定 native SDK：

```powershell
$env:OHOS_NATIVE_SDK="<path-to-openharmony-native-sdk>"
. .\scripts\ohos-env.ps1
rustup target add aarch64-unknown-linux-ohos
cargo build --target aarch64-unknown-linux-ohos -p tswn_core --bin tswn-cli --release --features no_debug
```

Linux/macOS 下对应：

```bash
export OHOS_NATIVE_SDK=/path/to/openharmony/native
. ./scripts/ohos-env.sh
rustup target add aarch64-unknown-linux-ohos
cargo build --target aarch64-unknown-linux-ohos -p tswn_core --bin tswn-cli --release --features no_debug
```

脚本会配置 `aarch64-unknown-linux-ohos` 目标需要的 linker、AR、CC、CXX、CFLAGS、CXXFLAGS、CMake toolchain 和 `CARGO_ENCODED_RUSTFLAGS`。其中 rustflags 会继续带上本项目 release 路径需要的 `-Z mutable-noalias=no`。

### 编译器包装器

新增 `ohos-clang` / `ohos-clangxx` 包装器：

- `ohos-clang.ps1`
- `ohos-clangxx.ps1`
- `ohos-clang.sh`
- `ohos-clangxx.sh`

它们读取 `OHOS_NATIVE_SDK`，然后调用 SDK 中的 clang / clang++，自动附加 `-target aarch64-linux-ohos`、`--sysroot` 和 `-D__MUSL__`。

### build_all 聚合包支持

`scripts/build_all.py` 新增 `--include-ohos-cli`，可以在完整发布包中额外构建鸿蒙 / OpenHarmony CLI：

```powershell
$env:OHOS_NATIVE_SDK="<path-to-openharmony-native-sdk>"
uv run scripts/build_all.py --release --clean --include-ohos-cli
```

默认产物名为：

```text
cli/bin/tswn-cli_alpha_0_3_10_aarch64_unknown_linux_ohos_unsigned.bin
```

这是未签名 ELF 二进制。直接调试时通常可放到 `/data/local/tmp/` 执行；如果部署位置或目标环境要求签名，需要按 OpenHarmony 侧要求另行签名。本仓库的聚合脚本只负责产出 unsigned binary，不做签名。

### 已验证的 full build

本版本已验证完整聚合构建路径可产出：

- Windows CLI：`tswn-cli_alpha_0_3_10.exe`

## 行为与兼容性

### 对 core 调用方

底层 `Runner` / `PreparedRunner` 仍保持原有定位。新增 `cli_api` 是为了给高层包装层提供统一入口，不强制已有调用方迁移。

需要注意的是，胜者队伍映射现在有更直接的 API。此前从 `winner` player id 自行反推输入队伍的调用方，可以考虑改用 `winner_team_index()` / `winner_team_indices()`。

### 对 CLI 用户

`namer-pf --mode`、`--precision`、`bench win-rate --double-plus` 都是新增能力。默认行为尽量保持兼容。`bench win-rate` 的输入解析修复会让包含 `\n` 或 overlay 的输入更符合预期。

### 对包装层用户

Python、WASM、C-API 新增大量高层 helper。它们的共同目标是“用包装层调用时得到与 CLI 一致的结构化结果”。旧的底层对象接口仍可继续使用。

### 对 OHOS / 鸿蒙用户

OHOS CLI 当前只提供 unsigned binary 构建和打包。能否在目标设备上直接运行，取决于部署路径、设备策略和签名要求。用于调试时建议优先放到 `/data/local/tmp/`；正式部署前需要按目标环境签名。

## 总结

`0.3.10` 是一次面向“多端可用性”和“发布工程化”的版本：

- core 新增 `cli_api`，把 CLI / Python / WASM / C-API 的高层语义收敛到同一套逻辑；
- DIY / OL 的 summon、shadow、zombie 导出继续补全，复杂衍生体 roundtrip 更可靠；
- namer-pf 和 win-rate 的多个边界继续向 JS 参考实现靠拢；
- Python replay / live helper 补齐，服务端回放构建更直接；
- WASM 和 C-API 补齐 CLI 对齐接口；
- OpenHarmony / OHOS 交叉编译进入正式构建文档和聚合包流程。

如果从 `0.3.9` 升级，最明显的变化是包装层能力大幅补齐，以及发布包现在可以同时带 Windows、Linux、OHOS、WASM、Python、C-API 和 Openbox 产物。
