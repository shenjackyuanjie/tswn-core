# tswn_core 0.3.4 更新说明（RunUpdate delay 对齐）

> `0.3.4` 主要同步战斗日志播放节奏相关字段，目标是让 `tswn_core` / `tswn_wasm` 的 delay 行为对齐 `fast-namerena/branch/original/md5.js` 的混淆版实现。
>
> 本次以混淆版 `md5.js` 为准；未混淆 Dart 源只作为辅助理解，不作为最终行为依据。

---

## 1. 改动目标

之前 Rust 侧 `RunUpdate::new()` 默认 `delay0=0`、`delay1=0`，wasm 示例页则按 `delay1 || delay0` 把 delay 挂到当前消息之后。

混淆版 `md5.js` 的实际行为不同：

- `RunUpdate` 默认构造携带 `delay0=1000`、`delay1=100`
- 渲染器处理可见 update 时，等待时间是 `max(update.delay0, 上一条可见 update.delay1)`
- 每帧起始的上一条 delay 为 `1800`
- normal 播放还会按 `sqrt(角色数 / 2)` 缩放等待时间
- `RunUpdateWin` 进入胜利渲染前有 `3000ms` delay

本次更新把 core 的 delay 字段和 wasm show 的使用方式统一到这套语义。

## 2. 变更清单

### 2.1 `RunUpdate` 默认 delay 对齐

| 文件 | 变更 |
| --- | --- |
| `crates/tswn_core/src/engine/update.rs` | 新增 `DEFAULT_DELAY0_MS=1000`、`DEFAULT_DELAY1_MS=100` |
| `crates/tswn_core/src/engine/update.rs` | `RunUpdate::new()` 默认写入 `delay0=1000`、`delay1=100` |

`new_dummy()` 和 `new_newline()` 仍保持 `delay0=0`、`delay1=0`，因为占位帧和换行帧不产生可见等待。

### 2.2 特殊 update delay 补齐

| 场景 | 文件 | delay |
| --- | --- | --- |
| 伤害反弹 | `player/skill/skl/reflect.rs` | `delay0=1500` |
| 护身符抵挡死亡 | `player/skill/skl/reraise.rs` | `delay0=1500` |
| 召唤亡灵 | `player/skill/skl/zombie.rs` | `delay0=1500` |
| 雷击每段伤害 update | `player/skill/act/thunder.rs` | `delay0=300` |
| 埼玉觉得有点饿 | `player/boss/saitama.rs` | `delay1=2000` |

这些值来自混淆版 `md5.js` 中的 `T.f(...)` / `aX.aK(...)` 构造参数和后续字段赋值。

### 2.3 wasm `total_delay` 语义调整

`crates/tswn_wasm/src/fight.rs` 新增 `playback_total_delay()`，按混淆版渲染器规则累加可见 update 等待时间：

```text
每帧开始：next_wait = 1800
每条可见 update：
  wait = max(update.delay0, next_wait)
  total_delay += wait
  next_wait = update.delay1
```

如果当前 frame 已结束，还会额外加入 `RunUpdateWin` 的 `3000ms` 等待。

`RoundFrame.total_delay` 的注释也已更新：它现在表示“原始未按角色数缩放的可见等待总和”。

### 2.4 wasm show 播放逻辑调整

| 文件 | 变更 |
| --- | --- |
| `examples/show-render.js` | chunk delay 改为按 `max(delay0, next_wait)` 计算 |
| `examples/show-render.js` | 结束帧 winner chunk 使用 `3000ms` delay |
| `examples/show.js` | normal 模式改为先等待再渲染当前 chunk |
| `examples/show.js` | normal 模式按 `sqrt(角色数 / 2)` 缩放 delay |
| `examples/show-replay.js` | 更新 `total_delay` 注释 |

fast / turbo 模式保留示例页自己的播放策略：fast 仍把整帧压缩到固定 `40ms`，turbo 仍为零延迟。

---

## 3. 兼容性与影响

- `RunUpdate::new()` 的默认 delay 字段发生变化；如果下游直接读取 `delay0/delay1`，会看到从 `0/0` 变为 `1000/100`。
- 战斗判定、伤害结算、目标选择、状态顺序不在本次改动范围内。
- wasm 输出的 `total_delay` 语义更接近原始页面播放节奏；如果下游把它当成简单字段累加，需要改为按“播放总等待时间”理解。
- show 示例 normal 模式播放速度会更接近原始名字竞技场；大量角色时会按原始缩放规则加速。
