# tswn_core 0.2.17 更新说明（改进 cqp 帮助、batch-rate 终端输出，并统一 benchmark 并行阈值）

> `0.2.17` 这一版继续打磨 `tswn-cli bench batch-rate` / `bench cqp` 的可用性：一方面把帮助信息写清楚，让 `cqp` 和 `batch-rate` 的对应关系直接出现在 help 里；另一方面给批量胜率任务补上更适合长批次观察的终端输出，包括可选阈值过滤、总进度、总体 ETA、最近 5 个选手窗口 ETA，以及只在交互终端启用的进度条刷新。同时，这轮还把当前 benchmark 路径的并行启用阈值统一到了 `n >= 100`，让中小样本批次能更早利用多线程。
>
> 这轮主要有五件事：
>
> 1. 明确 `bench cqp` 与 `bench batch-rate` 是同一个命令；
> 2. 新增 `--min-wr/-m`，允许按万分比阈值过滤终端显示结果；
> 3. 为 `batch-rate` 补上总进度、总体 ETA 和最近 5 个样本窗口 ETA；
> 4. 对进度条增加交互终端检测，避免污染重定向和日志输出；
> 5. 将当前 benchmark 路径的并行启用阈值统一到 `n >= 100`。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

`batch-rate` 这条命令原本已经能做批量胜率计算，但这轮继续使用下来，仍然有三个明显痛点：

- `bench cqp` 虽然已经是别名，但 help 里没有直接把两者关系写透；
- 长批次运行时只能等结果逐条刷出来，缺少整体进度和剩余时间感知；
- 进度条如果直接写到非交互输出里，会把清行控制符和重绘痕迹一起带进日志。

因此，`0.2.17` 的目标很明确：

- 把 `cqp` 和 `batch-rate` 的对应关系在 help 里写明白；
- 允许直接按胜率阈值过滤终端显示，减少大批量筛选时的噪音；
- 给长批次任务补上能看、能估时、也不会污染重定向输出的进度反馈。
- 顺手把当前 benchmark 的串并行切换阈值统一下来，避免中小样本批次过早退回串行。

---

## 2. 主要改动

## 2.1 帮助信息明确标出 `cqp` 别名

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli.rs`
- `crates\tswn_core\src\bin\tswn_cli\args.rs`

这一版把 `batch-rate` 相关帮助信息补全到了两个层级：

- 顶层 CLI 模块说明里，明确写出 `bench batch-rate` 的别名是 `bench cqp`；
- `clap` 子命令文案里，明确说明 `cqp` 与 `batch-rate` 是同一个命令的两个名字，功能完全相同。

同时也补了更直接的示例，例如：

```bash
tswn-cli bench cqp -l targets.txt -p players.txt -n 10000 -m 5000
```

这样后续无论是看源码顶部示例、`bench --help`，还是直接看 `bench batch-rate --help` / `bench cqp --help`，都能马上知道：

- `cqp` 不是另一条独立命令；
- 它只是 `batch-rate` 的可见别名；
- 两者参数、行为和输出都完全一致。

这轮还顺手让 `tswn-cli` 的 `clap` 顶层 help 显式带上 crate 版本号，方便直接确认当前二进制版本。

## 2.2 新增 `--min-wr/-m` 过滤终端显示

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\args.rs`
- `crates\tswn_core\src\bin\tswn_cli\bench.rs`

`bench batch-rate` 新增了：

- `--min-wr <N>`
- `-m <N>`

这个参数接受 `0..10000` 之间的整数，按万分比表示一个最低胜率阈值。

例如：

- `-m 5000` 表示只在终端显示达到该阈值的结果；
- `-m 8000` 表示只保留更高胜率的候选项；
- 不传时则保持原来的终端输出行为。

这里有一条刻意保留的约束：

- 该参数只影响终端展示；
- `--out-file` 写出的 JSONL 结果仍然保持完整输出，不因阈值过滤而丢记录。

这样既能满足“终端只看达标项”的筛选需求，也不会影响后续离线分析或二次处理。

## 2.3 为长批次任务补上总进度和双 ETA

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\bench.rs`

`batch-rate` 现在会在终端上显示一条批量进度条。

进度条本身按：

- `选手组 × 靶子组`

这一层的对局数推进，也就是每算完一个“某选手组 vs 某靶子组”的胜率摘要，就前进一步。

但 ETA 则按“完成了多少个选手组”来估算，分成两种口径：

- 总体预计：用已完成选手组的整体平均耗时估算剩余时间；
- 滑动预计：只看最近 5 个选手组的耗时，估算当前阶段下的剩余时间。

这样做的目的，是同时保留两种信息：

- 总体预计更稳定，不容易因为短时抖动大起大落；
- 滑动预计更灵敏，更接近“最近这几组”的真实速度。

对于长批次筛选任务，这通常比只有单一 ETA 更有用。

## 2.4 进度条只在交互终端下刷新

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\bench.rs`

这轮给 `BatchProgress` 增加了交互终端检测：

- 只有当 `stderr` 本身是交互终端时，才启用进度条刷新；
- 非交互场景下，不再输出清行和重绘控制符；
- 完成提示也跟随一起关闭，避免重定向日志里混入额外的 ANSI 序列和半截进度条痕迹。

这意味着下面几种场景会更干净：

- `2> log.txt`
- 管道转交给其他进程处理 stderr
- CI/脚本环境下收集命令输出

也就是说，这次不是“去掉进度条”，而是把它限制在真正适合刷新的交互终端环境里。

## 2.5 统一 benchmark 并行启用阈值

涉及文件：

- `crates\tswn_core\src\win_rate.rs`
- `crates\tswn_core\src\bin\tswn_cli.rs`
- `crates\tswn_core\src\bin\tswn_cli\bench.rs`
- `crates\tswn_core\src\bin\tswn_cli\fight.rs`

这一条和 help / 终端输出属于不同方向，但确实是这轮一起落进当前版本里的独立小型性能调整。

旧逻辑里有两套较保守的并行切换条件：

- `prepared_win_rate(...)` 在 `n < 2000` 时固定退回串行；
- CLI 单组评分和 raw `!test!` 单组评分路径也沿用了同样的 `2000` 阈值。

现在统一改成：

- 可用 worker 数大于 `1` 且 `n >= 100` 时，就允许进入并行路径；
- `prepared_win_rate(...)` 与 CLI 评分 benchmark 使用同一阈值；
- 因此在 `100..1999` 场这个区间里，prepared win-rate 和单组评分 benchmark 不会再被过早压回串行。

直接受影响的入口包括：

- `tswn-cli bench auto` 的单组评分分支
- `tswn-cli bench win-rate`
- `tswn-cli bench group-win-rate`
- `tswn-cli bench batch-rate`
- raw `!test!` 单组评分
- raw `!test!` 双组胜率
- `tswn_capi` 的 prepared win-rate 接口
- `tswn_py` 的 prepared/group win-rate 包装

其中 `tswn_capi` / `tswn_py` 仍然只通过 `prepared_win_rate(...)` 这一侧受影响；评分阈值统一只发生在当前 CLI 路径。

## 3. 使用示例

示例命令：

```bash
tswn-cli bench batch-rate -l targets.txt -p players.txt -n 10000
tswn-cli bench cqp -l targets.txt -p players.txt -n 10000 -m 5000
tswn-cli bench cqp -l targets.txt -p players.txt -n 10000 --verbose --perf
tswn-cli bench batch-rate -l targets.txt -p players.txt -o result.jsonl --force
```

适用场景包括：

- 固定一批靶子，快速筛掉平均胜率明显不达标的候选项；
- 在交互终端里长时间跑批量任务时，随时查看总进度和剩余时间；
- 在脚本或重定向输出场景里保留干净日志，而不被进度条刷新干扰。
- 在 `100..1999` 场这类中小样本 benchmark 下，更早利用并行 worker 缩短等待时间。

本轮完成后，建议至少验证：

- `cargo check -p tswn_core --bin tswn-cli`
- `tswn-cli bench --help`
- `tswn-cli bench batch-rate --help`
- 交互终端下实际执行一条 `bench cqp` 样例，确认进度条、总体 ETA 和滑动 ETA 正常刷新
