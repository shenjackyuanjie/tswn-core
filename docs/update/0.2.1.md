# tswn_core 0.2.1 更新说明（单线程优化 + bench 模式拆分 + Python 接口同步）

> 本次是 0.2.0 后的性能补强版本，重点响应两个诉求：
>
> 1. 单线程下希望尽量接近多线程性能  
> 2. CLI benchmark 需要明确区分“默认并行”和“单线程模式”

---

## 1. 改动目标

0.2.0 已把多线程 `--win_rate` 提升到很高吞吐，但单线程仍明显慢于并行。  
这次目标是：

- 在**不改变战斗语义**前提下继续压缩单线程耗时；
- 把 CLI benchmark 模式拆分清晰，避免“默认行为不透明”；
- 核心公开接口新增后，`tswn_py` 同步补齐，保持绑定层可用性。

---

## 2. CLI benchmark 模式拆分

涉及文件：`crates\tswn_core\src\bin\tswn_cli.rs`

### 新增命令

- 单线程：
  - `--bench-st`
  - `--bench-raw-st`
  - `--bench-file-st`
  - `--win_rate_st`

- 默认并行（保持原命令）：
  - `--bench`
  - `--bench-raw`
  - `--bench-file`
  - `--win_rate`

### 设计说明

- 新增 `BenchThreadMode`（`Parallel` / `SingleThread`）；
- 默认 bench 走并行模式；
- 单线程模式通过独立命令显式触发，便于做回归对比；
- 线程数可由 `TSWN_BENCH_WORKERS` 覆盖（`TSWN_WINRATE_WORKERS` 仍兼容）。

### 结果

- CLI 行为更清晰：用户能直接对比同一输入在并行/单线程下的表现；
- 大规模评分模式（单组 bench）也接入并行执行，不再只优化胜率模式。

## 3. 单线程热点优化（核心）

### 3.1 Runner 预构建模板缓存

涉及文件：`crates\tswn_core\src\engine\runners.rs`

#### 原问题

`Runner::new_from_groups_with_seed` 在高频调用时，会重复做：

- 玩家实例化
- 队内 upgrade
- `build` 和 boss 初始化

这些步骤与 seed 无关，却每局重复，导致单线程初始化成本偏高。

#### 方案

- 引入按分组内容缓存的**预构建模板**：
  - 缓存 `PreparedRunnerTemplate`（预构建玩家 + 基础名字集合）
  - 后续同组输入只做“按 seed 注入随机初始化流程”，复用模板数据
- 保留原随机流顺序与分组排序逻辑，确保行为一致性。

#### 效果

- 单线程 `--win_rate_st` 初始化成本显著降低；
- 多线程路径也受益（每个 worker 减少重复冷启动开销）。

---

### 3.2 SkillStorage 热路径去 clone

涉及文件：`crates\tswn_core\src\player\skill\store.rs`

#### 原问题

多个流程函数（`pre_action/post_action/pre_defend/...`）每次都会 `clone()` 一份 key 列表后再遍历，造成频繁小分配与拷贝。

#### 方案

- 改为按索引直接遍历已有 key 向量，不再临时 clone；
- 行为保持不变，只减少运行时开销。

#### 效果

- 高频战斗循环中的内存分配与拷贝下降；
- 对单线程尤其有帮助（并行时也有累计收益）。

## 6. 性能结果（代表性数据）

测试命令（release + no_debug，直接运行二进制）：

```powershell
.\target\release\tswn-cli.exe --win_rate_st aaa bbb 10000
.\target\release\tswn-cli.exe --win_rate aaa bbb 10000
```

代表结果：

- 单线程：约 `0.214s`（`21.4µs/场`）
- 多线程：约 `0.059s`（`5.9µs/场`）

相对改进：

- 单线程从约 `0.344s` 降到约 `0.214s`（显著下降）
- 多线程继续维持在高吞吐区间，并可稳定低于 `0.07s`

> 说明：单线程仍难与并行完全等价，这是计算资源上限导致的客观差异；本次已尽量压缩“算法外”开销（重复构建、分配、拷贝）。
