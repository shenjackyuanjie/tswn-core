# tswn_core 0.2.10 更新说明（CLI clap 化、子命令重构与 win-rate 对齐 JS 基准）

> `0.2.10` 一方面把 `tswn-cli` 从超长手写参数分支改成了 `clap` 子命令结构，另一方面也顺手把 `bench win-rate` 这条链路重新对齐到了 `fast-namerena/branch/latest/md5.js` 的实际 `win_rate` 行为。
>
> 这一版的关键点不是单纯“换个 help 库”或者“加一个新命令”，而是把 CLI 入口、benchmark 参数、group 对战接口，以及 JS 基准对齐过程中挖出来的两个隐藏 bug 一起收掉。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

这一轮的目标最开始其实很明确：

- 把 `tswn-cli` 的参数解析从手写逻辑迁到 `clap`；
- 把过长的 CLI 文件拆模块；
- 把旧的平铺 flag 入口收敛成更稳定的子命令；
- benchmark 线程配置不再依赖环境变量；
- 补一个“固定目标组 vs 多个对手组”的胜率接口。

但做到一半时，又碰上了另一个更关键的问题：

- Rust `bench win-rate` 和 JS `md5.js::win_rate` 在同样名字下给出的结果并不一致；
- 表面看像是 `rq=6 / --keep-rq` 差异；
- 继续往下查，才发现真正的问题不止一个，而且有一部分还在 Rust benchmark 自己的统计和高速推进路径里。

所以 `0.2.10` 最终变成了一版“CLI 结构重构 + 基准语义纠偏”的组合更新。

## 2. 主要改动

## 2.1 `tswn-cli` 参数解析迁移到 `clap`

涉及文件：

- `crates\tswn_core\Cargo.toml`
- `crates\tswn_core\src\bin\tswn_cli.rs`
- `crates\tswn_core\src\bin\tswn_cli\args.rs`

### 本次改动

- 使用 `cargo add clap -F derive` 引入 `clap`；
- 原本 CLI 中超长的手写参数匹配逻辑，改成了 `derive` 风格的命令树；
- help 文案不再靠手写大块 `after_help` 字符串拼接，而是改成 `clap` 原生 help 输出。

### 效果

这带来的收益不是“代码看起来现代一点”这么简单，而是：

- CLI 参数定义和实际分发逻辑终于收敛成同一份结构；
- 新增命令时不用再继续往一个超长分支里塞特判；
- `--help`、参数错误提示、必填参数校验都交给 `clap` 统一处理；
- 后续继续给 benchmark / icon / fight 加参数时，维护成本会小很多。

### 效果

这让 CLI 从“堆 flag 的工具”变成了真正按功能组织的接口：

- fight、benchmark、icon 不再互相缠在一块；

- 新增 `group-win-rate` 时不需要继续污染顶层命令空间。

## 2.3 benchmark 参数统一：移除 worker 环境变量，改成显式参数

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\args.rs`
- `crates\tswn_core\src\bin\tswn_cli\bench.rs`

### 本次改动

- 移除 benchmark 线程数相关环境变量入口：
  - `TSWN_BENCH_WORKERS`
  - `TSWN_WINRATE_WORKERS`
- 统一改成显式参数：
  - `-t, --thread <N>`
  - `--single-thread`
- 原先独立的 `perf` 逻辑并入 benchmark，统一用 `--perf` 开关控制输出。

### 效果

这主要解决的是“结果复现被隐藏环境状态污染”的问题：

- 以前 benchmark 结果可能被外部环境变量悄悄影响；
- 现在线程配置直接体现在命令行里，复现实验更直观；
- `bench auto / bench win-rate / bench group-win-rate` 的计时输出也统一了。

## 2.4 新增 `bench group-win-rate`

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\args.rs`
- `crates\tswn_core\src\bin\tswn_cli\bench.rs`

### 本次改动

- 新增 `bench group-win-rate` 子命令；
- 支持：
  - `--target <TARGET>`
  - `-a, --against <GROUP>`（可重复）
- 每个 `against` 都既可以是单人输入，也可以是多行组输入；
- 输出每组对局胜率、平均胜率和汇总胜率。

### 效果

这个接口主要是为了支持“固定目标组，批量打多组对手”的评估场景。

相比一条条手工跑：

- 命令结构更稳定；
- 输出更适合批量比较；
- 也避免了用户在外面再自己包一层脚本来汇总。

## 2.6 `bench win-rate` 重新对齐 JS `md5.js::win_rate`

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\bench.rs`
- `crates\tswn_core\src\engine\runners.rs`
- `crates\tswn_core\src\player\eval_name.rs`
- `sby_test.md`
- `fast-namerena\branch\latest\md5.js`

### 排查过程里先确认的事实

这一轮最重要的中间结论有三个：

1. Rust CLI `fight --raw "...seed:i@!"` 和 JS `runner.fight("...seed:i@!")` 在很多显式 seed 的单局对战上是一致的。
2. `--keep-rq` 对应的其实是“普通 seeded fight”语义，而不是 JS 官方 `win_rate` 语义。
3. JS 官方 `win_rate` 不是简单地拿 `seed:i@!` 连跑 1 万局，它走的是 `ProfileWinChance` 自己的一套调度。

### 最终修掉的三个问题

#### 1. benchmark 判胜对象搞错了

Rust 原先在 `bench win-rate` 里，会把“排序后 world 中的第一队”当成输入第一队来统计胜率。

这在内部排序发生变化时会直接把胜负算反。

修正后：

- 改成按原始输入组对应的 roster 统计胜者；
- 不再依赖 `world.teams.first()` 来猜测输入第一队。

#### 2. `run_to_completion()` 的高速路径和正常对局不等价

Rust 原先的 benchmark 高速路径直接自己 tick，空回合与停止条件并不完全等价于 CLI 正常 `main_round()` 推进。

这会导致一种很隐蔽的现象：

- 普通 CLI `fight` 单局看起来和 JS 一致；
- 但 benchmark 批量跑时，同一个 seed 序列却会出现胜者分叉。

修正后：

- `run_to_completion()` 改为复用 `main_round()`；
- 空回合判定与正常 fight 路径保持一致；
- benchmark 不再因为高速路径语义差异而偏离普通对局结果。

#### 3. 默认 `bench win-rate` 的 seed 调度不对

JS 官方 `ProfileWinChance.O()` 的语义不是：

- `seed:0@!`
- `seed:1@!`
- `seed:2@!`

而是：

- 第 1 局不带 seed；
- 第 2 局起从 `seed:33554431@!` 开始递增。

Rust 默认 `bench win-rate` 之前没有复刻这一点，所以即使底层单局 fight 没问题，最终胜率也会和 JS 官方接口偏掉一截。

修正后：

- 默认 `bench win-rate` 使用 JS `ProfileWinChance` 的 seed 调度；
- `--keep-rq` 继续保留普通 `fight + seed:i@!` 语义，方便做对照。

### 最终效果

修完后，默认模式和 `--keep-rq` 两条语义已经分开且都可解释：

- 默认 `bench win-rate`：对齐 JS 官方 `md5.js::win_rate`
- `--keep-rq`：对齐 JS 普通 `fight + seed:i@!`

实测样例：

- `SB` vs `LJ`
  - Rust 默认：`50.15%`
  - JS 官方 `win_rate`：`50.15%`
  - Rust `--keep-rq`：`48.54%`
  - JS seeded `fight` 汇总：`48.54%`

- `喘际瞬爆@昀澤` vs `蕾蒂·怀特洛可-65HEZHB264LFPFQ@Squall`
  - Rust 默认：`49.00%`
  - JS 官方 `win_rate`：`49.00%`
  - Rust `--keep-rq`：`47.92%`
  - JS seeded `fight` 汇总：`47.92%`

这说明这一轮修完后：

- Rust 默认 benchmark 已重新贴齐 JS 官方胜率接口；
- `--keep-rq` 也仍然保留为有意义的“普通对战语义”对照模式。

## 2.9 分发与打包补充

涉及文件：

- `scripts\build_capi.py`
- `scripts\build_all.py`
- `scripts\README.md`
- `crates\tswn_capi\examples\README.md`

### 本次改动

这一轮除了 CLI / benchmark / JS 对齐之外，也顺手把对外分发链路整理了一遍，目标是把“能 build”进一步推进到“能直接发包”。

#### 1. 新增 `tswn_capi` 分发脚本

新增：

- `scripts\build_capi.py`

该脚本会：

- 现场构建 `tswn_capi`
- 收集 `include/`
- 收集 `lib/` 下的动态库和伴生产物
- 收集 `examples/`
- 生成分发目录内的 `README.txt` 与 `MANIFEST.txt`

这样对外提供 C-API 时，不再需要手工从 `target\release` 和 crate 目录里拼装文件。

#### 2. 新增聚合打包脚本

新增：

- `scripts\build_all.py`

该脚本会统一处理三类产物：

- `capi`：现场构建并打包
- `cli`：现场构建并打包
- `py`：只收集当前已经存在的 wheel / 分发产物，不在脚本里强行现场构建

最终会生成：

- 一个带版本信息的 bundle 目录
- 一个对应的 zip 压缩包

默认命名会自动组合各组件版本，例如：

- `tswn_core_0_2_10_capi_0_1_0_py_0_1_8_bundle`

这样外部分发时，不同版本包之间更容易区分。

#### 3. CLI 分发文件名带版本号

聚合打包时，CLI 不再直接保留裸文件名：

- `tswn-cli.exe`

而是整理成带 `tswn_core` 版本号的文件名，例如：

- `tswn-cli_alpha_0_2_10.exe`

这样在外部目录里放多版 CLI 时，不容易被覆盖，也更方便人工识别。

#### 4. Python 分发补充 examples

聚合打包时，`py/` 目录除了收集现有 wheel 之外，也会一并收集：

- `crates\tswn_py\examples\`

这样用户拿到分发包后，不仅有 wheel，还能直接看到 Python 侧的最小调用示例。

#### 5. README 与脚本说明补充

本次还同步补充了两类说明文档：

- `scripts\README.md`
- bundle 顶层 `README.txt`

前者用于说明各构建脚本的用途、典型命令和输出位置；后者则补充 bundle 内各组件版本、主要产物路径以及使用说明。

#### 6. `tswn_capi` 补充跨编译器示例

`tswn_capi` 的分发 README 与 `examples\README.md` 中，都补充了跨编译器的示例编译方式，包括：

- MSVC / `cl`
- clang（Windows）
- gcc（Windows / MinGW 风格）
- clang / gcc / cc（Linux/macOS）

同时也补充了 Windows 下的运行说明，例如：

- 如果当前 shell 中没有 `cl`，可先执行 `start-vs-pwsh.ps1`
- 运行 `.exe` 时，若找不到 `tswn_capi.dll`，可以先把 DLL 复制到 `.exe` 同目录

其中以下两条方式已经在当前打包结果上做过实际验证：

- MSVC / `cl` 编译 `examples\version_and_error.c`
- clang（Windows）显式传入 `lib\tswn_capi.dll.lib` 编译 `examples\version_and_error.c`

并且编译出的 MSVC 示例程序在把 `tswn_capi.dll` 放到可执行文件同目录后，已经可以正常运行并输出版本与错误信息。

### 效果

这部分改动解决的不是核心算法问题，而是“对外可交付性”问题：

- `capi` 不再需要手工拼分发目录
- `cli` / `capi` / `py` 可以统一整理成一个带版本信息的 zip
- C-API 不再只有头文件和 DLL，本次连 examples 与编译说明也一起对外提供
- 包内 README 不再只是罗列目录，而是给出更接近“拿到包后怎么用”的操作提示

换句话说，`0.2.10` 除了把 CLI 结构与 JS 基准语义整理干净，也把这套结果往“可直接发给别人使用”推进了一步。

## 2.10 `release + no_debug` 与 Bun 的 `win-rate` 性能对比

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\bench.rs`
- `crates\tswn_core\src\engine\runners.rs`
- `crates\tswn_core\src\player\skill\act\haste.rs`
- `crates\tswn_core\src\player\skill\act\iron.rs`
- `fast-namerena\branch\latest\md5.js`

### 测试条件

- 机器：Ryzen 7 5800X（8C16T）
- Rust 构建：`cargo build -p tswn_core --bin tswn-cli --release --features no_debug`
- 对比样例：
  - `aaa` vs `bbb`
  - `喘际瞬爆@昀澤` vs `蕾蒂·怀特洛可-65HEZHB264LFPFQ@Squall`
- 场次：`100000`

### Rust `no_debug` 结果

#### 样例 1：`aaa` vs `bbb`

- 单线程：`2.749s`
  - `27.5µs/场`
  - `36379 场/s`
- 默认多线程：`0.493s`
  - `4.9µs/场`
  - `202824 场/s`
- 指定 `-t 16`：`0.675s`
  - `6.8µs/场`
  - `148147 场/s`

#### 样例 2：`喘际瞬爆@昀澤` vs `蕾蒂·怀特洛可-65HEZHB264LFPFQ@Squall`

- 单线程：`4.247s`
  - `42.5µs/场`
  - `23545 场/s`
- 默认多线程：`0.843s`
  - `8.4µs/场`
  - `118609 场/s`
- 指定 `-t 16`：`0.889s`
  - `8.9µs/场`
  - `112514 场/s`

### Bun `md5.js::win_rate` 结果

#### 样例 1：`aaa` vs `bbb`

- Bun 官方 `win_rate("!test!\n\naaa\n\nbbb", 100000)`：`12.674s`
  - `126.7µs/场`
  - `7890 场/s`

#### 样例 2：`喘际瞬爆@昀澤` vs `蕾蒂·怀特洛可-65HEZHB264LFPFQ@Squall`

- Bun 官方 `win_rate("!test!\n\n喘际瞬爆@昀澤\n\n蕾蒂·怀特洛可-65HEZHB264LFPFQ@Squall", 100000)`：`15.524s`
  - `155.2µs/场`
  - `6442 场/s`

### 可以直接得出的结论

1. Rust `no_debug` 单线程已经明显快于 Bun

样例 1：

- `12.674 / 2.749 ≈ 4.61x`

样例 2：

- `15.524 / 4.247 ≈ 3.66x`

也就是说，在不依赖并行的情况下：

- 对轻一点的 `aaa/bbb`，Rust 单线程大约快 `4.61x`；
- 对更重一点的长中文名样例，Rust 单线程也仍然大约快 `3.66x`。

2. Rust 默认多线程相对单线程还有进一步放大

样例 1：

- `2.749 / 0.493 ≈ 5.57x`

样例 2：

- `4.247 / 0.843 ≈ 5.04x`

这意味着在这台 `5800X` 上，当前 `win-rate` workload 用默认自动线程配置时，大致能拿到 `5x` 左右的单机加速。

3. 强行 `-t 16` 反而不如默认自动线程

- `aaa/bbb`：默认 `0.493s`，`-t 16` 为 `0.675s`
- 长中文名样例：默认 `0.843s`，`-t 16` 为 `0.889s`

这说明当前这条 benchmark 的瓶颈并不是“线程越多越好”，而是已经开始受到调度、同步和缓存竞争的影响。

换句话说：

- 在 `5800X (8C16T)` 上，默认自动 worker 数比手工拉满 `16` 线程更合适；
- 这也说明现在的默认线程策略至少对 `win-rate aaa bbb 10w` 这类场景是合理的。

4. Rust 默认多线程相对 Bun 的总倍率

样例 1：

- `12.674 / 0.493 ≈ 25.71x`

样例 2：

- `15.524 / 0.843 ≈ 18.41x`

也就是在这组样例里：

- `aaa/bbb`：Rust `no_debug` 默认多线程约快 `25.71x`
- 长中文名样例：Rust `no_debug` 默认多线程约快 `18.41x`

名字更长、更重的样例会拉低绝对吞吐，但不会改变总体趋势：

- Rust 单线程依然明显快于 Bun；
- Rust 默认多线程依然显著快于 Bun；
- 默认自动线程数依然优于强行 `-t 16`。

### 顺手做的清理

在做 `no_debug` 构建验证时，也顺手清掉了两处只在旧调试探针里使用的 warning 来源：

- `crates\tswn_core\src\player\skill\act\haste.rs`
- `crates\tswn_core\src\player\skill\act\iron.rs`

这两处改动里，`haste.rs` 还额外删除了保留下来的 `haste_post_action` 调试输出代码；`iron.rs` 则把只在旧调试打印里使用的未使用参数一起整理掉。

最终 `release + no_debug` 构建已恢复到无 warning 状态。

## 3. 影响 / 对齐点 / 兼容性

除 CLI 与 benchmark 行为层面的收敛外，这一版还新增了分发层面的几个直接变化：

- 仓库现在提供了 `capi` 专用分发脚本；
- 仓库现在提供了统一的聚合打包脚本；
- 聚合包默认会带版本化命名；
- `py` 分发不再只有 wheel，也会附带 examples；
- `capi` 分发包和 examples 目录中都带了编译 / 运行说明。

这意味着 `0.2.10` 不只是“开发时可用”，也更适合作为一版可直接发给外部调用方的打包基线。

`0.2.10` 没有再引入新的 crate API breaking change，但 CLI 和 benchmark 行为层面有几个很重要的收敛点：

- CLI 参数体系正式切换到 `clap`；
- 顶层接口从平铺 flag 收敛到子命令结构；
- benchmark 线程数不再被环境变量偷偷污染；
- 默认 `bench win-rate` 现在和 JS 官方 `win_rate` 真正对齐；
- `--keep-rq` 明确保留为“普通 seeded fight 语义”的对照模式。

如果你主要把 `tswn-cli` 当作人手运行的工具，这一版最直接的收益是命令结构更清楚；
如果你主要把它当成 JS 基准对账器，这一版更关键的价值是默认 `win-rate` 终于不再“看起来差一点但说不清到底差在哪”。

## 4. 验证

本轮实际做过的关键验证包括：

- `cargo check -p tswn_core --bin tswn-cli`
- `cargo build -p tswn_core --bin tswn-cli --release --features no_debug`
- `python scripts/build_capi.py --release`
- `python scripts/build_all.py --release --clean`
- 使用 `start-vs-pwsh.ps1` 进入 Visual Studio Developer PowerShell 环境后，执行：
  - `cl /nologo /Iinclude examples\version_and_error.c /link /OUT:examples\version_and_error.exe lib\tswn_capi.dll.lib`
- 使用 clang（Windows）执行：
  - `clang -Iinclude examples/version_and_error.c lib/tswn_capi.dll.lib -o examples/version_and_error.exe`
- 将 `tswn_capi.dll` 放到生成的 `.exe` 同目录后，实际运行 `version_and_error` 示例，确认能够正常输出 ABI、版本号、默认 `eval_rq` 以及错误信息
- 多组样例下对比 Rust 与 JS：
  - 显式 `seed:i@!` 单局结果
  - `fight + seed:i@!` 1 万局汇总
  - JS 官方 `win_rate` 结果
- `aaa` vs `bbb`、`100000` 场下对比 Rust `release + no_debug` 与 Bun `md5.js::win_rate` 吞吐：
  - Rust 单线程：`2.749s`
  - Rust 默认多线程：`0.493s`
  - Rust `-t 16`：`0.675s`
  - Bun：`12.674s`
- `喘际瞬爆@昀澤` vs `蕾蒂·怀特洛可-65HEZHB264LFPFQ@Squall`、`100000` 场下对比 Rust `release + no_debug` 与 Bun `md5.js::win_rate` 吞吐：
  - Rust 单线程：`4.247s`
  - Rust 默认多线程：`0.843s`
  - Rust `-t 16`：`0.889s`
  - Bun：`15.524s`
- 重点对账样例：
  - `SB` vs `LJ`
  - `喘际瞬爆@昀澤` vs `蕾蒂·怀特洛可-65HEZHB264LFPFQ@Squall`

最终关键结果：

- 默认 `bench win-rate` 与 JS 官方 `md5.js::win_rate` 对齐；
- `--keep-rq` 与 JS 普通 `fight + seed:i@!` 对齐；
- `run_to_completion()` 不再和普通 fight 路径出现胜者分叉。
- `release + no_debug` 下，`aaa/bbb` 样例中 Rust 单线程约快于 Bun `4.61x`、默认多线程约快于 Bun `25.71x`；长中文名样例中 Rust 单线程约快于 Bun `3.66x`、默认多线程约快于 Bun `18.41x`。
