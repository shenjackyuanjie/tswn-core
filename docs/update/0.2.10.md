# tswn_core 0.2.10 更新说明（CLI clap 化、子命令重构与 win-rate 对齐 JS 基准）

> `0.2.10` 一方面把 `tswn-cli` 从超长手写参数分支改成了 `clap` 子命令结构，另一方面也顺手把 `bench win-rate` 这条链路重新对齐到了 `fast-namerena/branch/latest/md5.js` 的实际 `win_rate` 行为。
>
> 这一版的关键点不是单纯“换个 help 库”或者“加一个新命令”，而是把 CLI 入口、benchmark 参数、group 对战接口，以及 JS 基准对齐过程中挖出来的两个隐藏 bug 一起收掉。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

这一轮的目标最开始其实很明确：

- 把 `tswn-cli` 的参数解析从手写逻辑迁到 `clap`；
- 把过长的 CLI 文件拆模块；
- 把旧的平铺 flag 入口收敛成更稳定的子命令；
- benchmark 线程配置不再依赖环境变量；
- 补一个“固定目标组 vs 多个对手组”的胜率接口。

但做到一半时，又碰上了另一个更关键的问题：

- Rust `bench win-rate` 和 JS `md5.js::win_rate` 在同样名字下给出的结果并不一致；
- 表面看像是 `rq=6 / --keep-rq` 差异；
- 继续往下查，才发现真正的问题不止一个，而且有一部分还在 Rust benchmark 自己的统计和高速推进路径里。

所以 `0.2.10` 最终变成了一版“CLI 结构重构 + 基准语义纠偏”的组合更新。

---

## 2. 主要改动

## 2.1 `tswn-cli` 参数解析迁移到 `clap`

涉及文件：

- `crates\tswn_core\Cargo.toml`
- `crates\tswn_core\src\bin\tswn_cli.rs`
- `crates\tswn_core\src\bin\tswn_cli\args.rs`

### 本次改动

- 使用 `cargo add clap -F derive` 引入 `clap`；
- 原本 CLI 中超长的手写参数匹配逻辑，改成了 `derive` 风格的命令树；
- help 文案不再靠手写大块 `after_help` 字符串拼接，而是改成 `clap` 原生 help 输出。

### 效果

这带来的收益不是“代码看起来现代一点”这么简单，而是：

- CLI 参数定义和实际分发逻辑终于收敛成同一份结构；
- 新增命令时不用再继续往一个超长分支里塞特判；
- `--help`、参数错误提示、必填参数校验都交给 `clap` 统一处理；
- 后续继续给 benchmark / icon / fight 加参数时，维护成本会小很多。

---

## 2.2 CLI 改成子命令结构，并完成模块化拆分

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli.rs`
- `crates\tswn_core\src\bin\tswn_cli\args.rs`
- `crates\tswn_core\src\bin\tswn_cli\bench.rs`
- `crates\tswn_core\src\bin\tswn_cli\fight.rs`
- `crates\tswn_core\src\bin\tswn_cli\icon.rs`

### 本次改动

- 顶层命令收敛为：
  - `fight`
  - `bench`
  - `icon`
- `bench` 下细分为：
  - `bench auto`
  - `bench win-rate`
  - `bench group-win-rate`
- `icon` 下细分为：
  - `icon show`
  - `icon b64`
  - `icon save`
- 原本超长的 CLI 单文件按职责拆成 `args / bench / fight / icon` 四个模块。

### 效果

这让 CLI 从“堆 flag 的工具”变成了真正按功能组织的接口：

- fight、benchmark、icon 不再互相缠在一块；
- benchmark 相关参数都收口在 `bench` 子树里；
- 新增 `group-win-rate` 时不需要继续污染顶层命令空间。

---

## 2.3 benchmark 参数统一：移除 worker 环境变量，改成显式参数

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\args.rs`
- `crates\tswn_core\src\bin\tswn_cli\bench.rs`

### 本次改动

- 移除 benchmark 线程数相关环境变量入口：
  - `TSWN_BENCH_WORKERS`
  - `TSWN_WINRATE_WORKERS`
- 统一改成显式参数：
  - `-t, --thread <N>`
  - `--single-thread`
- 原先独立的 `perf` 逻辑并入 benchmark，统一用 `--perf` 开关控制输出。

### 效果

这主要解决的是“结果复现被隐藏环境状态污染”的问题：

- 以前 benchmark 结果可能被外部环境变量悄悄影响；
- 现在线程配置直接体现在命令行里，复现实验更直观；
- `bench auto / bench win-rate / bench group-win-rate` 的计时输出也统一了。

---

## 2.4 新增 `bench group-win-rate`

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\args.rs`
- `crates\tswn_core\src\bin\tswn_cli\bench.rs`

### 本次改动

- 新增 `bench group-win-rate` 子命令；
- 支持：
  - `--target <TARGET>`
  - `-a, --against <GROUP>`（可重复）
- 每个 `against` 都既可以是单人输入，也可以是多行组输入；
- 输出每组对局胜率、平均胜率和汇总胜率。

### 效果

这个接口主要是为了支持“固定目标组，批量打多组对手”的评估场景。

相比一条条手工跑：

- 命令结构更稳定；
- 输出更适合批量比较；
- 也避免了用户在外面再自己包一层脚本来汇总。

---

## 2.5 `fight` 输出追加 `win_idx`，保留 stdin 默认行为

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\fight.rs`

### 本次改动

- `fight` 继续保留“默认从 stdin 读取输入”的旧行为；
- 普通输出和 `--out-raw` 输出在结尾都追加 `win_idx=...`；
- 赢家索引基于原始输入顺序，而不是内部排序后的队伍顺序。

### 效果

这对手工看日志和做 case 对账都更方便：

- 可以直接知道胜者在原始输入里的位置；
- 不需要再把内部 `PlrId` 或排序后的 roster 反推回去。

## 2.6 `bench win-rate` 重新对齐 JS `md5.js::win_rate`

涉及文件：

- `crates\tswn_core\src\bin\tswn_cli\bench.rs`
- `crates\tswn_core\src\engine\runners.rs`
- `crates\tswn_core\src\player\eval_name.rs`
- `sby_test.md`
- `fast-namerena\branch\latest\md5.js`

### 排查过程里先确认的事实

这一轮最重要的中间结论有三个：

1. Rust CLI `fight --raw "...seed:i@!"` 和 JS `runner.fight("...seed:i@!")` 在很多显式 seed 的单局对战上是一致的。
2. `--keep-rq` 对应的其实是“普通 seeded fight”语义，而不是 JS 官方 `win_rate` 语义。
3. JS 官方 `win_rate` 不是简单地拿 `seed:i@!` 连跑 1 万局，它走的是 `ProfileWinChance` 自己的一套调度。

### 最终修掉的三个问题

#### 1. benchmark 判胜对象搞错了

Rust 原先在 `bench win-rate` 里，会把“排序后 world 中的第一队”当成输入第一队来统计胜率。

这在内部排序发生变化时会直接把胜负算反。

修正后：

- 改成按原始输入组对应的 roster 统计胜者；
- 不再依赖 `world.teams.first()` 来猜测输入第一队。

#### 2. `run_to_completion()` 的高速路径和正常对局不等价

Rust 原先的 benchmark 高速路径直接自己 tick，空回合与停止条件并不完全等价于 CLI 正常 `main_round()` 推进。

这会导致一种很隐蔽的现象：

- 普通 CLI `fight` 单局看起来和 JS 一致；
- 但 benchmark 批量跑时，同一个 seed 序列却会出现胜者分叉。

修正后：

- `run_to_completion()` 改为复用 `main_round()`；
- 空回合判定与正常 fight 路径保持一致；
- benchmark 不再因为高速路径语义差异而偏离普通对局结果。

#### 3. 默认 `bench win-rate` 的 seed 调度不对

JS 官方 `ProfileWinChance.O()` 的语义不是：

- `seed:0@!`
- `seed:1@!`
- `seed:2@!`

而是：

- 第 1 局不带 seed；
- 第 2 局起从 `seed:33554431@!` 开始递增。

Rust 默认 `bench win-rate` 之前没有复刻这一点，所以即使底层单局 fight 没问题，最终胜率也会和 JS 官方接口偏掉一截。

修正后：

- 默认 `bench win-rate` 使用 JS `ProfileWinChance` 的 seed 调度；
- `--keep-rq` 继续保留普通 `fight + seed:i@!` 语义，方便做对照。

### 最终效果

修完后，默认模式和 `--keep-rq` 两条语义已经分开且都可解释：

- 默认 `bench win-rate`：对齐 JS 官方 `md5.js::win_rate`
- `--keep-rq`：对齐 JS 普通 `fight + seed:i@!`

实测样例：

- `SB` vs `LJ`
  - Rust 默认：`50.15%`
  - JS 官方 `win_rate`：`50.15%`
  - Rust `--keep-rq`：`48.54%`
  - JS seeded `fight` 汇总：`48.54%`

- `喘际瞬爆@昀澤` vs `蕾蒂·怀特洛可-65HEZHB264LFPFQ@Squall`
  - Rust 默认：`49.00%`
  - JS 官方 `win_rate`：`49.00%`
  - Rust `--keep-rq`：`47.92%`
  - JS seeded `fight` 汇总：`47.92%`

这说明这一轮修完后：

- Rust 默认 benchmark 已重新贴齐 JS 官方胜率接口；
- `--keep-rq` 也仍然保留为有意义的“普通对战语义”对照模式。

### 效果

这让 `rq=6` 不再只是一个“猜出来的神秘兼容值”，而是和 JS `ProfileWinChance` 语义明确绑定。

## 3. 影响 / 对齐点 / 兼容性

`0.2.10` 没有再引入新的 crate API breaking change，但 CLI 和 benchmark 行为层面有几个很重要的收敛点：

- CLI 参数体系正式切换到 `clap`；
- 顶层接口从平铺 flag 收敛到子命令结构；
- benchmark 线程数不再被环境变量偷偷污染；
- 默认 `bench win-rate` 现在和 JS 官方 `win_rate` 真正对齐；
- `--keep-rq` 明确保留为“普通 seeded fight 语义”的对照模式。

如果你主要把 `tswn-cli` 当作人手运行的工具，这一版最直接的收益是命令结构更清楚；
如果你主要把它当成 JS 基准对账器，这一版更关键的价值是默认 `win-rate` 终于不再“看起来差一点但说不清到底差在哪”。

## 4. 验证

本轮实际做过的关键验证包括：

- `cargo check -p tswn_core --bin tswn-cli`
- `cargo build -p tswn_core --bin tswn-cli --release --features no_debug`
- 多组样例下对比 Rust 与 JS：
  - 显式 `seed:i@!` 单局结果
  - `fight + seed:i@!` 1 万局汇总
  - JS 官方 `win_rate` 结果
- 重点对账样例：
  - `SB` vs `LJ`
  - `喘际瞬爆@昀澤` vs `蕾蒂·怀特洛可-65HEZHB264LFPFQ@Squall`

最终关键结果：

- 默认 `bench win-rate` 与 JS 官方 `md5.js::win_rate` 对齐；
- `--keep-rq` 与 JS 普通 `fight + seed:i@!` 对齐；
- `run_to_completion()` 不再和普通 fight 路径出现胜者分叉。
