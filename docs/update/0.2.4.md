# tswn_core 0.2.4 更新说明（Charm 视角、`post_damage` 顺序与 linked minion 清理修正）

> 这版主要围绕一批 SBY 对账失败 case 收敛细节：不是大功能扩展，而是把目标选择、后处理顺序和召唤物清理顺序继续往 JS 行为上拉齐。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

`0.2.3` 之后，Rust 侧已经更容易做 raw 日志对账；接下来暴露出来的问题就更集中在**行为顺序本身**。

`0.2.4` 主要要解决四类误差：

- 魅惑链中“同队/异队”的判断视角仍有偏差；
- `post_damage` 的执行顺序还不够像 JS；
- owner 死亡时 linked minion 的清理顺序不稳定；
- 净化导致的状态取消文案在死亡边界上会多打一行。

这版可以看成一次典型的“顺序语义版修复”。

---

## 2. 主要改动

## 2.1 Charm 队伍比较改为基于有效队伍视角

涉及文件：`crates\tswn_core\src\player\skill\act\charm.rs`

此前 Rust 在处理魅惑重施时，判断“是否还是同队”的逻辑更接近静态原始队伍；但 JS / Dart 的语义是：

- 如果施法者自己也被魅惑，比较时应使用施法者**当前生效的 allyGroup**；
- 也就是说，比较基准不是“原始在哪个队”，而是“此刻在行为上被视为什么队”。

### 本次修正

- 读取施法者自身是否带有 `CharmState`；
- 若施法者已被魅惑，则比较其 charm 后的有效队伍；
- 若未被魅惑，才回退到原始队伍。

### 效果

这修正了链式魅惑、重魅惑场景下的同队判断偏差，使选目标、状态叠加和后续行为更接近 JS 基准。

---

## 2.2 引入 `post_damage_priority()`，统一 `post_damage` 排序语义

涉及文件：

- `crates\tswn_core\src\player\skill.rs`
- `crates\tswn_core\src\player\skill\store.rs`
- `crates\tswn_core\src\player\skill\act\assassinate.rs`

本版给 `SkillTrait` 增加了：

- `post_damage_priority()`

默认值为 `10000`，并允许具体技能覆盖。

### 为什么要加这个入口

JS 里很多后处理钩子的顺序依赖 `sortId / ga4()` 一类语义。Rust 之前虽然也有 `post_damage`，但缺少一个显式优先级层，导致不同技能在边界 case 下的顺序容易飘。

### 本次修正

- `SkillStorage` 在构建 `post_damage` 列表时，按优先级排序；
- `Assassinate` 显式返回 `i32::MAX`，确保在 `Hide` 之后执行。

### 效果

最直接的结果是：

- “潜行被识破”和“发动隐匿”这类输出顺序得到修正；
- `post_damage` 阶段的行为链更接近 JS；
- 外部扩展技能如果未来需要特殊顺序，也有了明确的覆写点。

## 2.3 linked minion 清理顺序改为队伍 roster 顺序

涉及文件：`crates\tswn_core\src\player\impl_runtime.rs`

此前 owner 死亡时，linked minion 的清理次序更接近 HashMap key 遍历顺序，这会带来两个问题：

- 顺序不稳定；
- `?1` 可能先于 `?0` 被清理，导致日志反序。

### 本次修正

- 优先使用 owner 所在队伍的 roster 顺序；
- 同队内按真实出现顺序清理 linked minion；
- 只有在缺失队伍上下文时才回退到排序后的全量 id 集合。

### 效果

owner 死亡时，linked minion 会先清 `?0`、再清 `?1`，日志和清理次序都更稳定，也更符合玩家的直觉。

---

## 2.4 净化取消文案只在“目标仍存活”时输出

涉及文件：`crates\tswn_core\src\player\impl_runtime.rs`

此前净化（disperse）在 on-damage 回调里清正向状态时，即使目标本次伤害已经把 HP 打空，也可能继续输出“某状态被打消”的取消文案。

JS 侧更接近的语义是：

- 如果目标已经进入本次死亡结算边界，这类取消文案不应再插入。

### 本次修正

- 只有 `alive()` 且 `hp > 0` 时，才输出清状态相关消息。

### 效果

这消除了死亡边界前的一行多余文案，让净化与死亡结算的日志顺序更接近基准实现。

## 3. 影响 / 对齐点 / 兼容性

`0.2.4` 没有引入新的 Rust API breaking change。

不过行为层面会有明显变化：

- 魅惑链中的同队判断更准确；
- `post_damage` 相关技能的触发顺序会变化；
- linked minion 的清理与日志顺序更稳定；
- 净化在死亡边界上的输出更干净。

如果你在做黄金日志或 SBY 对账，这版是会改变结果的；如果你只关心 crate 接口本身，则迁移成本几乎为零。
