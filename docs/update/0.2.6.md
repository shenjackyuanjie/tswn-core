# tswn_core 0.2.6 更新说明（同级状态顺序、召唤物同步与调试追踪补强）

> 这版继续围绕剩余少量 failed case 收敛语义差异：重点不在新增机制，而是在“同层状态谁先结算”“召唤物什么时候算真正进场”“调试时能不能更快定位偏差”这几件事上继续贴近 JS。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

`0.2.5` 之后，大批明显的战斗尾部和状态生命周期问题已经收敛，剩下的问题更集中在顺序与同步细节：

- 同优先级状态在 Rust 里仍可能退化成不稳定顺序；
- summon / shadow 这类同回合新生成实体，在 sync 前后的可见性仍有少量偏差；
- `Hide` 对“真实同队存活者”的理解还没完全摆脱旧 fallback；
- 大样本调试虽然已经能做，但追踪粒度还不够细。

所以 `0.2.6` 的重点是：把状态顺序、召唤物同步和调试入口再压一轮。

---

## 2. 主要改动

## 2.1 同级状态后处理顺序改为按注册顺序稳定执行

涉及文件：

- `crates\tswn_core\src\player\state.rs`
- `crates\tswn_core\src\player\skill\act\iron.rs`

此前 Rust 在多个状态拥有相同 `post_action` / `post_damage` 优先级时，仍可能退化成按 tag 或 HashMap 迭代顺序执行；这和 JS 中“同层按注册先后进入链表”的语义不一致。

### 本次修正

- `PlayerStateStore` 开始记录状态注册顺序；
- 同优先级状态排序时，先按 priority，再按 registration order；
- 状态被移除后重新加入，会拿到新的注册顺序；
- `IronState` 的 `post_action` 优先级也调整到与 `SlowState` 同层，便于通过注册顺序复现 JS 的结束顺序。

### 效果

像 `Slow` / `Iron` 这类同层 timer 状态，Rust 侧不再固定成错误顺序，debug / release 下也更稳定。

---

## 2.2 技能与状态边界行为继续对齐

涉及文件：

- `crates\tswn_core\src\player\skill\act\ice.rs`
- `crates\tswn_core\src\player\skill\act\thunder.rs`
- `crates\tswn_core\src\player\skill\skl\hide.rs`

### 本次修正

- 冰冻术在施法者处于 charge 激活时，会正确追加额外 `frozen_step`；
- 雷击术循环中，补上对施法者 active、目标存活与目标 frozen 状态的检查，避免多打一段或多消耗 RNG；
- `Hide` 不再在缺失真实 alive group 时回退到 clan 级别统计；
- 同一 action 内新生成的 shadow，会先按 JS 语义进入“当前可见 roster”再参与后续判定，避免 `round_pos` 和调度链漂移。

### 效果

这类修正的核心是：

- 哪些实体此刻算 active；
- 哪些新实体此刻已经“看得见”；
- 哪些 fallback 在 JS 里其实不存在。

对齐后，`Hide`、`Thunder`、shadow 同步这类很容易引发 RC4 偏移的路径会稳定很多。

---

## 2.3 召唤物名称、固定槽位与 owner 死亡时清理继续修复

涉及文件：

- `crates\tswn_core\src\bin\tswn_case_miner.rs`
- `crates\tswn_core\src\bin\tswn_cli.rs`
- `crates\tswn_core\src\engine\engine_core.rs`
- `crates\tswn_core\src\player\skill\act\summon.rs`

### 本次修正

- 同一 root owner 重复召唤 Summon 时，会复用已有 `?n` trace 名称槽位，不再每次都被当成一个全新编号实体；
- owner 在同回合死亡时，尚未 sync 进 world 的 linked pending minion 也会被立即标记死亡；
- Summon 被吞噬、继承或 merge 后，固定技能槽位的语义继续对齐 JS；
- 相关测试同时覆盖了 pending shadow 和 fixed skill slot 场景。

### 效果

这让 CLI / case miner / 测试里的“同一个召唤物到底是不是同一个对象”更稳定，也让 owner 死亡与 minion 清理之间的生命周期更接近基准实现。

---

## 2.4 调试追踪继续补强

涉及文件：

- `crates\tswn_core\src\debug.rs`
- `crates\tswn_core\src\player\impl_runtime.rs`
- `crates\tswn_core\src\player\skill\store.rs`
- `crates\tswn_core\src\player\state.rs`

### 本次补强

- `TSWN_DEBUG_ACTION` 不再要求与 `id_name()` 完全相等，允许按包含关系命中目标；
- action / `post_action` / `post_damage` / death 等关键阶段增加了更细的 actor / id / rc4 追踪；
- 这让 `?n` 召唤物、shadow、linked minion 等过去不太好精确命中的对象更容易被直接追踪。

### 效果

当剩余分歧只剩个位数时，问题往往不再是“能不能发现差异”，而是“能不能快速把差异压到某一步”。

`0.2.6` 的调试增强，主要就是为这件事服务。

---

## 3. 影响 / 对齐点 / 兼容性

`0.2.6` 依然没有引入新的 Rust API breaking change，但行为层面会继续变化：

- 同级状态的结算顺序更稳定；
- `Hide` / `Thunder` / 冰冻 / shadow 同步的边界行为会变化；
- summon 的追踪命名和固定槽位表现更稳定；
- owner 死亡时 pending minion 的清理更接近 JS。

如果你在做 case miner 对账或读 raw trace，这版的价值会很明显；
如果你只从高层调用 crate，外部接口基本没有迁移压力。

## 4. 验证

本版主要依据以下提交整理：

- `05d6597`（冰冻术 charge 加成与雷击循环条件）
- `ad112f6`（同级状态后处理顺序）
- `6793f0e`、`1b2fa9d`、`54ab0e9`、`02c2044`（召唤物与同步修复）
- `6ea9390`（隐匿判定与调试追踪补强）

简版 changelog 中记录的验证结果为：

- `cargo test -p tswn_core` 全量通过（159 passed）
- 固定 SBY 大样本对比中，`diff_failures` 从 `13` 降至 `6`
- `ts_failures = 0`、`rust_failures = 0`

整理时也重点核对了：

- 同优先级状态是否改为按注册顺序执行；
- summon / pending minion / shadow 在 sync 前后的可见性是否收敛；
- `Hide` 是否完全切到真实 alive group；
- 调试输出是否已经覆盖 action / state / death 关键路径。
