# tswn_core 0.2.16 更新说明（对齐 md5.js 的最小 trim 语义）

> `0.2.16` 这一版主要修正名字解析里一个容易造成 JS/Rust 分叉的基础问题：此前 Rust 把一组来自 `md5.js` 的“可 trim 空白字符”直接当成了 `name/team` 的非法字符集合使用，导致像 `U+3000` 这样的字符在 JS 里通常只是被边界裁剪或保留，而在 Rust 里却会直接构造失败。
>
> 这轮先做最小对齐，不去完整重写 `parse_names` 的全部怪异细节，先处理当前最容易引发分叉的三件事：
>
> 1. 停止把 JS trim 字符当作 `name/team` 的 reject 字符；
> 2. raw 输入按 JS `\s` 规则裁掉每行尾部空白；
> 3. `+weapon` 后半段按 `md5.js trim_name` 的边界语义裁剪。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 问题背景

此前 `tswn_core` 里有一个 `filter_char`，它被直接用于 `Player::new_and_init(...)` 的 `name/team` 校验。

这带来两个问题：

- 一是语义错位：`md5.js` 里的对应表主要服务于 trim，不是“只要出现就非法”；
- 二是实现细节还有边界偏差：Rust 用了半开区间写法，和 JS 显式列举的码点集合并不完全相同。

最直接的表现就是：

- JS 里 `U+3000` 常常只是边界空白；
- Rust 里它此前会触发 `InvalidTextInName` / `InvalidTextInTeam`。

这会让很多本应可构造、可继续参与战斗的输入，在 Rust 里提前失败。

---

## 2. 本轮改动范围

这一版只做最小对齐，明确不去一次性复刻 JS `parse_names` 的全部前导空白怪异语义。

本轮已对齐的部分：

- `name/team` 不再因为 `U+3000`、`133` 同类 trim 字符而直接报非法字符；
- raw 输入分组前会按 JS `\s` 规则裁掉每一行尾部空白；
- `+weapon` 后半段会按 JS `trim_name` 风格裁掉两端可 trim 空白，并处理 `133` 扩展边界裁剪；
- 内部位置的 `U+3000` 之类字符会被保留，不再误删。
- 顺手清理了当前 workspace 的 clippy warning，避免这轮解析修正落地后又留下新的工程噪音。

本轮刻意未做的部分：

- 暂不模拟 JS `parse_names` 中 `^\s+[:@]*\s* -> " "` 的前导空白替换语义；
- 暂不重写整套名字分词/结构解析流程；
- 暂不清理 `PlayerError::InvalidTextInName/Team` 这两个旧错误类型，只是停止在当前路径继续误用。

---

## 3. 具体实现

涉及文件：

- `crates\tswn_core\src\player\utils.rs`
- `crates\tswn_core\src\player\impl_attr.rs`
- `crates\tswn_core\src\player\impl_ctor.rs`
- `crates\tswn_core\src\engine\runners.rs`
- `crates\tswn_core\src\player\test\basic.rs`
- `crates\tswn_core\src\engine\test\mod.rs`

### 3.1 拆分 trim helper

新增了两套明确分工的 helper：

- `is_js_regex_space(ch)`
  - 用于对齐 JS `\s`
  - 主要服务于 raw 行尾空白裁剪
- `is_js_trim_name_char(ch)`
  - 用于对齐 `md5.js` 的 `check_str_legeal`
  - 主要服务于 `trim_name` 风格处理

同时新增：

- `trim_js_line_end(s)`
- `trim_js_name_like(s)`

这样后续继续修名字解析时，至少不会再把“trim 集合”和“reject 集合”混成一个函数。

### 3.2 停止误用 trim 字符做 reject

`Player::new_and_init(...)` 里原先会直接检查：

- `team.chars().any(filter_char)`
- `name.chars().any(filter_char)`

然后返回：

- `InvalidTextInTeam`
- `InvalidTextInName`

这一版已经移除这段误用逻辑。

因此，像 `ab　cd` 或 `red　team` 这样的内部全角空格，不再因为“看起来像空白”就在构造阶段直接失败。

### 3.3 对齐最小解析链路

`Runner::split_namerena_into_groups(...)` 现在会在分组前先按 JS `\s` 规则处理每一行的尾部空白。

这意味着：

- 行尾 `U+3000` 不再残留到名字里；
- 只含可裁剪尾部空白的行能更接近 JS 地退化成空行，用于分组；
- 但行首空白仍先保留，不主动模拟 JS 那个前导空白替换怪逻辑。

`Player::new_from_namerena_raw(...)` 也同步做了两件事：

- 先裁掉整行末尾的 JS 空白；
- 解析 `+weapon` 时，对 weapon 后半段应用 `trim_js_name_like(...)`。

此外，和 JS 一样，在拆下 `+weapon` 后，左侧主串也会再做一次尾部空白裁剪，避免出现：

- `mario@red　+　fire　`

这种输入里 team 仍残留尾部全角空格的问题。

## 4. 回归测试

这一版补了几组最关键的回归：

- `U+3000` 在 `name/team` 内部时不再误报非法字符；
- `mario@red　+　fire　U+0085　` 这类输入会被正确解析成：
  - `name = mario`
  - `team = red`
  - `weapon = fire`
- `Runner::split_namerena_into_groups(...)` 会在分组前先裁掉行尾 `U+3000`；
- helper 本身也有单测覆盖 `U+3000`、`8233`、`65279`、`133` 等关键码点差异。

本轮实际执行的定向验证命令：

```bash
cargo fmt --package tswn_core
cargo test -p tswn_core player_raw_ -- --nocapture
cargo test -p tswn_core trim_js_ -- --nocapture
cargo test -p tswn_core split_namerena_groups -- --nocapture
cargo +nightly fmt
cargo clippy --workspace --all-targets
```

其中后两条也已经实际执行通过；这轮顺手把 `tswn_core` 和 `tswn_ds3` 的现存 clippy warning 一并收敛到了干净状态。

---

## 6. 后续建议

如果后面继续向 `md5.js` 靠齐，下一步最值得处理的是：

- `parse_names` 的前导空白特殊替换语义；
- 更完整的 raw 名字结构解析一致性；
- 基于真实 case 的回归比对，而不只是单元测试级别的 trim 行为确认。

但这几项都会比 `0.2.16` 明显更大，这次先把最核心、最容易误伤合法输入的 trim/reject 混用问题切开。
