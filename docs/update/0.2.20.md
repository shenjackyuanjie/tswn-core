# tswn_core 0.2.20 更新说明（继续回收 win-rate 性能回退）

> `0.2.20` 这一版的目标很单一：在 `0.2.19` 已经把 `3v3v3 + maxcase=4000` 的大样本 diff 收口到 `failed case = 0` 之后，继续把 `bench win-rate` 路径里最明显的性能回退往回收。
>
> 这轮真正保留下来的改动，集中在三条已经验证过 correctness 的热路径：
>
> - dynamic `pre_action`
> - `Hide` 的 same-action alive 视图
> - 一批主动技能里的 alive-group 长度查询
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

`0.2.19` 已经把这一轮 correctness / parity 风险最大的部分收住了：

- `cargo test -p tswn_core` 能稳定通过
- `tests\sqp5900.txt`、`3v3v3`、`max-cases-per-mode=4000` 口径下，`failed case = 0`

但当时按 `docs\performance.md` 的固定口径补跑 benchmark 后，`0.2.19` 相对 `0.2.14` 仍然有稳定回退，尤其多线程更明显。

因此，`0.2.20` 的出发点不是再扩大语义修复范围，而是：

1. 只保留能确认不打坏 parity 的性能改动；
2. 优先改那些覆盖范围大、行为容易证明等价的热路径；
3. 对 mixed-signal 的优化直接撤回，不把 speculative 复杂度带进版本里。

---

## 2. 主要改动

## 2.1 `SkillStorage` 的 dynamic `pre_action` 热路径继续收紧

涉及文件：

- `crates\tswn_core\src\player\skill\store.rs`
- `crates\tswn_core\src\player\impl_runtime.rs`

这轮把 `pre_action` 相关路径继续压了一遍：

- 去掉 `pre_action()` 主循环里的 `self.pre_action.clone()`
- 减少循环中的重复 `contains()` / lookup
- 把 dynamic `pre_action` 的 membership 独立成哈希集，保留 `Vec` 作为顺序视图
- `impl_runtime` 在 skill act 后直接复用当前 skill 上已知的 dynamic pre_action 状态，不再重新查一遍 store

核心目标是减少：

- 每轮 pre_action 遍历时的额外 clone
- 动态启停 pre_action 时的线性存在性检查
- action 热路径上的重复查表

这部分属于“行为不变、只减少重复工作”的优化。

## 2.2 `Hide` 改成借引用 + iterator + `SmallVec`

涉及文件：

- `crates\tswn_core\src\engine\storage.rs`
- `crates\tswn_core\src\player\skill\skl\hide.rs`

`Hide` 这一轮继续保留了已经验证过的一组低风险优化：

- 不再 clone alive group / roster group
- `Storage` 新增 iterator 风格的：
  - `iter_pending_spawn_ids_for_owner(...)`
  - `iter_pending_revival_ids_for_group(...)`
- `Hide` 改用 borrowed group 视图和 `SmallVec<[PlrId; 8]>` 聚合 alive candidates

这样既保留了 `0.2.19` 已经对齐好的 same-action 可见性语义，也去掉了几段在 post_damage 热路径里反复创建中间 `Vec` 的成本。

## 2.3 alive-group 长度查询改成基于 team index 的 O(1) 路径

涉及文件：

- `crates\tswn_core\src\engine\storage.rs`
- `crates\tswn_core\src\player\skill\act\assassinate.rs`
- `crates\tswn_core\src\player\skill\act\berserk.rs`
- `crates\tswn_core\src\player\skill\act\charm.rs`
- `crates\tswn_core\src\player\skill\act\curse.rs`
- `crates\tswn_core\src\player\skill\act\disperse.rs`
- `crates\tswn_core\src\player\skill\act\exchange.rs`
- `crates\tswn_core\src\player\skill\act\half.rs`
- `crates\tswn_core\src\player\skill\act\slow.rs`

之前这一批主动技能在 smart target score 里都会反复做：

```rust
alive_group_containing(target).map(|group| group.len()).unwrap_or(0)
```

它的真实成本是：

- 先在 `alive_groups` 上做一次线性扫描
- 找到组后再取长度

现在 `Storage` 新增了：

- `alive_group_len_containing(...)`

这条 helper 直接基于已有 `player_group` / team index 反查 alive_group，再确认 actor 确实还在该组里，语义和旧实现保持一致，但把这条高频查询从“扫整份 alive_groups”降成了“定点取组 + 常数级确认”。

由于它覆盖了 8 个主动技能的目标打分，所以这轮收益比单独抠某一个 skill 更稳定。

## 3. 撤回的尝试

这轮并不是所有尝试都保留了。

其中一组比较典型的是：

- `Protect` / `Counter` 的 group clone 消除

这条线在局部 spot-check 时看起来有正信号，但在后续 A/B 复测里表现成 mixed signal，没有形成“两个样本都稳定更快”的净收益，因此最终撤回，没有带进 `0.2.20`。

这也是本轮一个刻意坚持的原则：

- **没有证据证明稳定收益的优化，不留。**

---

## 4. 结果与 benchmark

当前建议继续参考 `docs\performance.md` 的固定口径：

- `--release --features no_debug`
- `tswn-cli bench win-rate ... --perf`
- `100k` 与 `1M`
- 单线程 / CLI 默认多线程

这轮同步回文档的结果是：

| 样本 | 单线程 `100k` | 多线程 `100k` | 单线程 `1M` | 多线程 `1M` |
|---|---:|---:|---:|---:|
| `aaa` vs `bbb` | `1.887s` | `0.291s` | `18.438s` | `2.784s` |
| `复杂样本` | `2.891s` | `0.471s` | `28.488s` | `3.749s` |

和 `0.2.14` 基线相比：

- 单线程当前慢约 `1.6%~2.3%`
- 多线程当前慢约 `7.9%~10.7%`

和上一轮 `0.2.19` 文档口径相比，这一版已经把回退继续收窄了一截，尤其 `1M` 口径更明显：

- `aaa` `1M` 单线程：`19.006s -> 18.438s`
- `aaa` `1M` 多线程：`2.836s -> 2.784s`
- `复杂样本` `1M` 单线程：`29.064s -> 28.488s`
- `复杂样本` `1M` 多线程：`3.886s -> 3.749s`

也就是说，`0.2.20` 当前可以明确确认的是：

- **correctness 继续稳住了**
- **大样本 parity 继续维持 `failed case = 0`**
- **性能回退继续缩小了，但多线程仍没完全回到 `0.2.14`**

## 5. 当前 git 修改摘要

当前真正保留下来的改动，可以概括成 4 组：

1. **dynamic `pre_action` 热路径**
   - clone / contains / redundant lookup 精简
   - membership set 化

2. **`Hide` / same-action alive 视图**
   - iterator helper
   - borrowed group
   - `SmallVec` 聚合

3. **alive-group 计数路径**
   - 新增 `alive_group_len_containing(...)`
   - 8 个高频主动技能统一切换到 O(1) 路径

4. **文档与 benchmark 口径同步**
   - `docs\performance.md`
   - 本文档
   - `CHANGELOG.md`
