# tswn_core 0.2.5 更新说明（战斗结束中断、状态时序与对账路径继续收敛）

> 这版开始明显进入“SBY 差异收尾期”：大方向不是再扩功能，而是继续把战斗结束边界、状态钩子顺序、友军视角和若干技能生命周期往 JS / `md5.js` 语义上压。
>
> 对应简版条目见：`crates\tswn_core\CHANGELOG.md`

---

## 1. 改动目标

`0.2.4` 之后，Rust 侧已经补上了 raw 日志、更细的顺序控制入口，以及一批明显的 `post_damage` / Charm 偏差。

接下来暴露出来的问题更集中在四类：

- 战斗其实已经结束，但 Rust 还在继续执行本回合尾部状态链；
- `Hide` / `Charm` / `Curse` 等状态对“当前友军是谁、当前还活着的是谁”的理解仍和 JS 有细差；
- 召唤物分摊伤害、技能升级、复活等生命周期边界还有少量顺序偏差；
- 大样本对账越来越依赖更低成本的调试与构造路径。

所以 `0.2.5` 可以看成一次“语义继续对齐 + 调试/热路径顺手收口”的版本。

---

## 2. 主要改动

## 2.1 战斗结束与后处理时序继续对齐

涉及文件：

- `crates\tswn_core\src\engine\engine_core.rs`
- `crates\tswn_core\src\player\impl_runtime.rs`
- `crates\tswn_core\src\player\skill\act\assassinate.rs`
- `crates\tswn_core\src\player\skill\act\curse.rs`
- `crates\tswn_core\src\player\skill\skl\hide.rs`

这版首先继续收口“本回合其实已经结束，但尾部逻辑还在跑”的问题。

### 本次修正

- 当一次行动已经决出胜负时，当前 actor 的 `recover / newline / post_action` 尾链不再继续执行；
- `Assassinate` 在强制行动前会主动清掉已死亡目标的陈旧锁定，避免状态驱动的动作模式继续被旧目标绑住；
- `CurseState` 的 `post_defend` 优先级改回与 JS `ga4()` 语义一致的默认层级，不再过早插入防御链；
- `post_defend` 中铁壁破裂后的状态刷新与诅咒附着时机继续后移，避免后续读取到旧快照；
- `Hide` 在 `post_damage` 阶段统计友军时，只再看真正仍然 `alive()` 的角色，避免刚死队友残留在快照里继续影响 RNG 消耗。

### 效果

- 战斗结束后的“尾巴日志”明显减少；
- `Curse` / `Defend` / `Iron` 一类防御链顺序更接近 JS；
- 刺杀、隐匿等依赖状态快照的行动路径更稳定。

---

## 2.2 队伍语义、状态生命周期与日志继续对齐

涉及文件：

- `crates\tswn_core\src\player\skill\skl\hide.rs`
- `crates\tswn_core\src\player\skill\skl\protect.rs`
- `crates\tswn_core\src\player\impl_runtime.rs`
- `crates\tswn_core\src\player\skill\act\accumulate.rs`
- `crates\tswn_core\src\player\skill\act\ice.rs`

### 本次修正

- `Hide` 的友军统计在存在魅惑时，改为基于魅惑后的有效队伍，而不是原始队伍；
- `revive_with_hp()` 不再顺手清掉冰冻状态，对齐 JS 的“复活只恢复 HP”语义；
- 治愈技能清除 `CurseState` 后，会补发“从诅咒中解除”的输出；
- 负面状态清除消息的输出次序改成稳定顺序，避免与基准实现的文案顺序漂移；
- `Protect` 在空魅惑队伍场景下不再多消耗一次 RC4；
- 聚气与蓄力的联动重新对齐：蓄力仍活跃时，聚气会补行动条并使用更高的临时攻击倍率；
- 冰冻术在智能选目标时，会下调已经处于冰冻状态目标的优先级，避免继续错误偏向同一目标；
- 净化后正向状态的清除顺序继续收口；
- 混合大小写 ASCII 名称的 `name_factor` 评分类别修正到与 JS 相同的哨兵类别；
- 链式魅惑开始显式继承已解析过的有效队伍视角，减少后续选目标、保护、隐藏路径的分叉。

### 效果

这批修正看上去零散，但本质上都在解决同一个问题：

- “当前队友是谁”
- “当前还活着的是谁”
- “状态解除/复活后的下一步视图应该是什么”

Rust 在这些边界上的理解更接近 JS 后，后续随机数轨迹和日志顺序也会更稳定。

## 2.3 召唤物、技能升级与运行时一致性修复

涉及文件：

- `crates\tswn_core\src\engine\storage.rs`
- `crates\tswn_core\src\player\skill\act\summon.rs`
- `crates\tswn_core\src\player\skill\store.rs`
- `crates\tswn_core\src\player\impl_runtime.rs`

### 本次修正

- 为使魔伤害分摊路径补上 `in_post_damage` 标记，owner 因分摊伤害死亡时，不再提前把 linked summon 走错死亡顺序；
- 吞噬 / 复制 / 合并后新启用的技能会补注册 proc，不再出现“技能已存在但钩子链没更新”的状态；
- 动作阶段会跳过 build 时未进入 action 列表的技能，避免升级后“倒插回当前动作队列”；
- 行动过程中发生的技能升级，不再被旧等级数据写回覆盖；
- `post_kill` 回调中对同一玩家的 `&mut Player` 别名问题被拆开处理，减少 release 构建下潜在的不一致行为。

### 效果

这些修正主要解决“代码表面看起来对了，但在 release / 连锁回调 / 召唤物联动里顺序不稳”的问题。

对账角度看，它让 Rust 侧更接近 JS 真正的生命周期链，而不是只在简单 case 下看起来一致。

---

## 2.4 调试与热路径整理

涉及文件：

- `crates\tswn_core\src\debug.rs`
- `crates\tswn_core\src\engine\runners.rs`
- `crates\tswn_core\src\bin\tswn_cli.rs`

### 本次整理

- 所有 `TSWN_DEBUG_*` 环境变量检查统一收口到 `debug` 模块；
- 启用 `no_debug` feature 时，这些检查可以在编译期被完全消除；
- `win_rate` 的 `rq` 行为改为显式配置，不再依赖容易串状态的隐式全局；
- 引入可复用的 `PreparedRunner` 预构建路径，减少大批量模拟时的重复构造开销；
- CLI 的 `--win_rate` / `--win_rate_st` 新增 `--keep-rq`，允许在兼容默认行为和普通对战语义之间切换。

### 效果

`0.2.5` 不只是修行为差异，也顺手把继续对账所需要的工具链整理得更适合高频回归：

- 调试开关更集中；
- release 热路径更干净；
- 胜率模拟与普通对战的 `rq` 语义也更容易区分。

---

## 3. 影响 / 对齐点 / 兼容性

`0.2.5` 没有引入新的 Rust API breaking change，但行为层面的变化很多：

- 战斗结束后的尾部状态日志会变少；
- `Hide` / `Charm` / `Curse` / `Protect` / revive 的边界行为会变化；
- 召唤物分摊伤害、技能升级、`post_kill` 链的执行顺序更稳定；
- `win_rate` 路径的兼容模式变得更显式。

如果你在做黄金日志、SBY 对账或 RC4 轨迹排查，这版会明显改变输出；
如果你只使用 crate 的基础接口，迁移成本依然不高。
