# tswn_core 0.3.3 更新说明（alive_group_count 对齐 / case miner 缓存修复）

> `0.3.3` 主要收束两类实际行为问题：
>
> 1. `alive_group_count` 改为对齐 JS `Engine.y.a.Q`，修复多队伍战斗里智能选目标的队伍计数漂移；
> 2. `tswn_case_miner` 的 TS trace 缓存签名改为同时跟踪 `md5.js` 与 `assets/**`，避免工具脚本改了却继续误命中旧缓存。
>
> 本批 commit 里还有一轮 `post_act_level` / clone 相关注释补齐；因为不改变运行逻辑，下文不单列。

---

## 1. 改动目标

Rust 侧之前把 `Storage::alive_group_count()` 实现成“遍历 `alive_groups` 后实时统计当前非空队伍数”。
但 JS 侧 `Engine.y.a.Q` 的语义并不是实时计数：

- 初始化时等于队伍数
- 某个队伍第一次被打空时递减
- 之后即使这个队伍通过 revive / addNew 重新出现存活成员，也不会补回

这会让 3 队及以上对局中，默认攻击与若干技能的 smart target scoring 在 Rust / JS 之间出现分叉。

另一个问题是 `tswn_case_miner` 的缓存签名原来只依赖 `out_md5.ts` 本体；如果旁边的 `md5.js` 或 `assets/**` 改了，旧 cache 仍可能命中，影响 TS / Rust 对账。

---

## 2. 变更清单

### 2.1 `alive_group_count` API 与同步路径调整

| 文件                    | 变更                                                                                                                                   |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| `engine/world_state.rs` | 新增 `alive_group_count` 字段与 `alive_group_count()` 访问器，按 JS 语义维护计数                                                       |
| `engine/storage.rs`     | `alive_group_count` 改为独立存储字段（当前实现为 `AtomicUsize`）；新增 `sync_alive_groups_owned_with_count(groups, alive_group_count)` |
| `engine/engine_core.rs` | runtime sync `alive_groups` 时显式携带 `world.alive_group_count()`                                                                     |
| `engine/runners.rs`     | 初始化 `WorldState` 后同步 `alive_groups` 时也携带初始计数                                                                             |

**行为变化：**

```text
旧行为：
  3v3v3 中某队被打空后，如果后来 revive / summon 回来，alive_group_count 会重新变回 3

新行为：
  一旦某队被打空，alive_group_count 递减；
  后续 revive / addNew 不再补回，保持与 JS Engine.y.a.Q 一致
```

### 2.2 智能选目标打分对齐 JS

`player/skill.rs` 默认 `score_target()` 不再临时扫描全体玩家并去重队伍头来推导“存活队伍数”，而是直接读取同步后的 `Storage::alive_group_count()`。

这带来两点效果：

- **语义对齐**：多队伍对局的 smart target scoring 与 JS 更接近
- **实现更直接**：去掉一次“扫全场 + 去重”的临时统计

这会影响默认攻击及一组依赖 `alive_group_count` / `alive_group_len_containing` 的智能选目标路径，尤其是 **3 队及以上** 且存在 **队伍被打空后又 revive / summon** 的样例。

### 2.3 `FireState` 初次挂载不再立即刷新状态

`player/skill/act/fire.rs` 中，首次给目标附加 `FireState` 时从 `set_state(...)` 改为 `set_state_no_update(...)`。

这样可以避免“刚上火状态时立即额外跑一次状态刷新”，减少与 JS 行为不一致的机会，也让火状态的首次附加路径更贴近当前 runtime 其他 delayed state 的处理方式。

### 2.4 `tswn_case_miner` 缓存失效条件增强

| 项目               | 旧行为                                | 新行为                                                 |
| ------------------ | ------------------------------------- | ------------------------------------------------------ |
| 缓存签名输入       | 只 hash `out_md5.ts`                  | 同时 hash `out_md5.ts`、同目录 `md5.js` 与 `assets/**` |
| 默认共享缓存根目录 | `<repo>/target/tswn_case_miner_cache` | `<repo>/tests/tswn_case_miner_cache`                   |

细节上，本次新增了：

- `md5_tool_signature(...)`
- `collect_signature_files(...)`
- `hash_file_for_signature(...)`

这意味着：

- 修改 `md5.js` 后，旧 TS trace cache 不会再被误复用
- 修改资源文件（如 `assets` 下内容）后，也能自动触发 cache miss
- 共享缓存目录更靠近测试资产目录，便于集中管理 case miner 相关缓存

### 2.5 测试覆盖补强

新增 / 扩充的回归测试包括：

- `engine/test/runner/large_66_70.rs`
  - `large_66`
  - `large_67`
  - `large_68`
- `bin/tswn_case_miner.rs`
  - `md5_tool_signature_tracks_md5_js_dependency()`

其中 `large_66~68` 主要用于覆盖近期定位到的 replay 分叉样例；`md5_tool_signature_tracks_md5_js_dependency()` 用于确认 `md5.js` 变更会实际改变缓存签名。

## 3. 兼容性与影响

- **无新的 breaking API 签名变更**；`0.3.2` → `0.3.3` 仍是 patch 级步进。
- 但 **`Storage::alive_group_count()` 的返回语义发生了有意调整**：
  - 旧版更接近“当前非空存活队伍数”
  - 新版改为“JS `Engine.y.a.Q` 兼容计数”
- 如果下游代码把它当成实时非空队伍数使用，在 revive / addNew 场景下可能会看到不同结果；这是为对齐 JS 而做的修正。
- `tswn_case_miner` 的默认共享缓存目录已迁移到 `tests/tswn_case_miner_cache`；如果你之前依赖旧目录，升级后第一次运行会出现一次正常的缓存冷启动。
- 现有环境变量 `TSWN_CASE_MINER_TS_CACHE_DIR` / `TSWN_CASE_MINER_BUN_CACHE_DIR` 覆盖方式保持不变。
