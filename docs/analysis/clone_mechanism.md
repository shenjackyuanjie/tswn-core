# 分身机制详解（CloneSkill）

## 概述

分身（`CloneSkill`, ID = 23）表面上生成一个与本体完全相同的克隆体，但实际上二者存在关键差异。整个过程是：**本体先发生衰减，再生成一个新的克隆体对象**。

---

## 1. 基本属性变化

### 本体（owner）

| 属性                         | 变化                             |
| ---------------------------- | -------------------------------- |
| 血量                         | × 0.5                            |
| 七围（攻/防/速/敏/魔/抗/智） | × 0.78                           |
| MP（蓝量）                   | **保留原值**，与正常使用技能无异 |
| 技能                         | 完全不变，衰减保留               |

### 克隆体（cloned）

| 属性       | 变化                                         |
| ---------- | -------------------------------------------- |
| 血量       | 继承本体的当前血量（至少为 1）               |
| 七围       | 拷贝本体的当前七围（含武器加成二次计入）     |
| MP（蓝量） | **重新初始化**：`智 / 2`（与普通新对象一致） |
| 技能       | **重新加载，大多数情况下不保留衰减**         |

---

## 2. 关键分歧点

### 2.1 MP 分歧

分身后本体和克隆体的 MP 来源不同：

- **本体**：保留使用技能前的 MP，没有重新初始化
- **克隆体**：作为一个全新的 Player 对象，走 `init_values()` 流程，`mp = wisdom >> 1`

因此分身后的本体和克隆体**MP 通常不相等**。

### 2.2 技能熟练度分歧

这是分身号对战局和评分影响最大的部分。

#### 正常情况：克隆体重新加载，不保留衰减

克隆体通过 `build_for_clone()` 重新构建技能，从 `name_base` 重新推导熟练度。这意味着克隆体的熟练度**回到 build 时的初始值**，本体则保留衰减后的值。

**实测例 1**：`涵虚不等式 PFVKEUPBU@TigerStar` 的幻术

```
初始熟练度: 92
使用多次幻术
衰减到: 50
分身后:
  本体幻术: 50（衰减保留）
  克隆体幻术: 92（重新加载，不保留衰减）
```

**实测例 2**：`G-Force #8W6dyD0vG@Shabby_fish` 的护符

```
初始熟练度: 66
使用多次护符
衰减到: 40
分身后:
  本体护符: 40（衰减保留）
  克隆体护符: 66（重新加载，不保留衰减）
```

---

## 3. 衰减下限（name_base 对名字强度的实际影响）

### 3.1 概念

部分技能的熟练度存在**衰减下限**，这个下限只存在于吃到以下加成的技能：

- **末尾主动技加成**（`boost_last()`：等级翻倍）
- **末尾座位加成**（slot 14/15 的被动 boost）

克隆体重新 build 时，如果本体已衰减到低于加成前的水平，克隆体不会恢复到完全初始值，而是恢复到**加成后但受当前 name_base 约束的上限**。

### 3.2 末尾主动技加成示例

以 `涵虚不等式 PFVKEUPBU@TigerStar` 的幻术为例：

```
name_base 解析:
  #08 幻术: 56 62 56 60 → 基础 46 / 加成后 92
    ↑ 末尾主动技：基础 46，boost_last() 翻倍 → 92
```

幻术 ID 为 24，它是该名字洗牌后排在最后的主动技能，吃到 `boost_last()` 翻倍。

**衰减行为**：

```
初始: 92（46 × 2）
衰减到 40（低于初始基础 46）

分身后:
  本体幻术: 40（衰减保留）
  克隆体幻术: 40 × 2 = 80（不是 92！）
```

克隆体重新 build 时：`boost_last()` 仍然生效，但翻倍的是**当前 name_base 推导出的基础值**（可能因衰减而降低），而非初始 92。

### 3.3 末尾座位加成示例

以 `G-Force #8W6dyD0vG@Shabby_fish` 的护符为例：

```
name_base 解析:
  #15 护符: 55 63 62 44 → 基础 34 / 加成后 66
    ↑ 末尾座位加成: slot 15, name_base[62]=32, name_base[63]=44
      基础 34 + boost min(32,44) = 34 + 32 = 66
```

护符在这里吃到 slot 15 的被动 boost（用 `name_base[62]` 和 `name_base[63]` 的较小值加成）。

**衰减行为**：

```
初始: 66（34 + 32）
衰减到 25（低于初始基础 34）

分身后:
  本体护符: 25（衰减保留）
  克隆体护符: 25 + 32 = 57（不是 66！）
```

克隆体重新 build 时：slot 15 的被动 boost 仍然生效，但基础值变了，且 `name_base[62]`/`name_base[63]` 也可能因衰减而变化。

### 3.4 末端座位加成的上限机制

`boost_level()` 的实际公式并非简单的 `level + boost`，而是：

```
boost_amount = min( name_base[src], name_base[src+1] )  // 原始加成量
actual_boost = min( boost_amount, skill.level() )        // 受当前等级上限
new_level    = skill.level() + actual_boost
```

即 **加成金额不能超过当前技能等级**。这个上限在普通 build 中通常不可见（原始加成量 ≤ 基础等级），但在 clone 构建时变得关键：

**实测例**：`以剑之名 #eEV76fMOX@Shabby_fish` 的护符

```
原始护符: base=41, boost=min(name_base[62],name_base[63])=41 → 41+41=82
衰减到: 21

分身后 (普通 clone):
  clamp: 41 → 21（截到 owner 当前值）
  boost: min(41, 21) = 21  ← 上限生效！
  → 21 + 21 = 42

分身后 (DIY clone):
  若不加上限: 21 + 41 = 62  ← 错误，多了 20
  加上限后:    21 + 21 = 42  ← 正确
```

**对 DIY 拟合的影响**：DIY clone 在应用 SlotBoost 时必须同样遵守 `boost.min(current)` 上限，否则会高估 clone 的衰减下限，导致普通 clone 和 DIY clone 的实际强度不同。

---

## 4. 对 DIY 拟合的影响

衰减下限的存在解释了为什么 **DIY 拟合长期无法完全精准**：

```
"SklShadow": 40 的可能来源：
  ├── 无任何加成，原始熟练度就是 40（无衰减下限）
  ├── 末尾主动技：20 × 2 = 40
  ├── 末尾座位加成：10 + 30 = 40
  ├── 末尾座位加成：11 + 29 = 40
  ├── 末尾座位加成：13 + 27 = 40
  └── ...
```

同一个最终熟练度 `40`，可能有完全不同的内部构成。这些差异在**普通对局**中不可见，但在**分身对局**中：

- 加成的存在与否决定衰减下限
- 加成的类型（翻倍 vs 加固定值）决定衰减后的恢复量
- 这些差异直接影响分身后克隆体的实际强度

因此仅靠 DIY 配一个最终熟练度值，无法还原分身号的完整行为。

---

## 5. 总结

|                  | 本体     | 克隆体                         |
| ---------------- | -------- | ------------------------------ |
| 血量             | × 0.5    | 继承本体当前值                 |
| 七围             | × 0.78   | 拷贝本体（含武器二次计入）     |
| MP               | 保留原值 | 重新初始化（智/2）             |
| 技能等级         | 保留衰减 | 重新 build 推导                |
| 有衰减下限的技能 | 衰减后值 | 恢复到衰减下限（非完全初始值） |

核心结论：**分身后的两个实体并非镜像，克隆体是一个"重新构造但受 name_base 约束"的新对象，而非本体的简单拷贝。**
