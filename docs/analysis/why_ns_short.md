# `why_ns` 到底干了啥

## 一句话

`why_ns` 是 JS `md5.js` 里的一个构造序号计数器。它有完善的**退款 + 重置**机制，门控阈值又极高（2048），**实际上无论普通 bench 还是 profile 几乎都触发不了**。

---

## 生命周期

```
a1()  构造玩家时:  读取当前值，然后 += 1
aZ()  挂进 roster: -= 1（退款）
O()/bE() 胜负输出: = 0（硬重置）
```

正常一局的执行流水：

```
构造玩家甲 → a1: why_ns = 1
挂 roster   → aZ: why_ns = 0    ← 退款
构造玩家乙 → a1: why_ns = 1
挂 roster   → aZ: why_ns = 0    ← 退款
...
胜负输出    → O(): why_ns = 0    ← 硬重置
```

每构造一次就退一次款，最后还要归零。**不可能跨轮累积。**

---

## 门控条件

`why_ns` 只有在 `abs(slot) >= 2048` 时才会生效——交换 RC4 state 前两个字节。

2048 次构造才触发一次，而在普通 bench 里一局最多构造几个玩家。所以这个计数器就算在递增，也**永远达不到阈值**。

---

## Profile 呢？

Profile 唯一的区别是：`ProfileMain.dZ()` 在真打之前会先做预构造（`choose_boss()`），这些预构造走 `a1()` 但不走 `aZ()` 退款，所以第一场真打的构造起点不是 0。

但要触发 2048 的门控，预构造得生成两千多个玩家才行——实际 profile 预构造只有个位数，**同样达不到阈值**。

`md5_score_probe.js` 里有个 `--reset-why` 参数，作用是在 `dZ()` 之后强行把 `why_ns = 0`。这个参数是用来排除"万一是 why_ns 的问题"的调试手段，**不代表它真的会触发**。

---

## 所以结论

| 场景 | 能否触发 2048 门控 | 实际影响 |
|------|-------------------|---------|
| 普通 bench | ❌ | 无 |
| namer-pf / score | ❌ | 无 |

`why_ns` 在 JS 里真实存在，生命周期也确实是文档描述的那样。但它生效的门槛太高，**不管普通对战还是 profile，实际都碰不到它**。

之前修掉的那几个 diff（计分语义、输入形状、TestEx、Clone、Summon、Iron）本来就不是 `why_ns` 的问题，现在可以确认了。
